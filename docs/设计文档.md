# Agent Bench 执行器设计文档

## 1. 目标

执行器接收云测任务，在本地完成固定主链：

```text
准备工程 -> Agent 修改代码 -> 编译验证 -> 上传产物 -> 上报结果
```

当前支持多个云测任务并发执行。每个 execution 有独立的任务目录、状态、进度和上报链；单个 execution 内部仍按固定主链串行执行。

## 2. 核心分层

```text
executor
  -> task_manager
    -> pipeline
      -> agent_runner
    -> cloud_api
    -> uploader
```

- `executor`：FastAPI 服务入口，注册云测任务接口，启动或检查 OpenCode 服务。
- `task_manager`：管理 execution 生命周期、等待队列、并发槽位、进度、上传和最终上报。
- `pipeline`：执行单任务主流程，准备 `original/` 和 `workspace/`，调用 Agent，编译验证，生成 diff。
- `agent_runner`：执行单次 Agent，会话通信、skill 挂载、SSE/HTTP 事件采集、question 自动应答都收口在这里。
- `cloud_api`：云测协议适配，负责请求模型、状态映射、上报载荷和 HTTP client。
- `uploader`：上传适配，目前主要是 AGC Cloud Storage。

## 3. 任务阶段

每个任务按以下阶段推进：

- `pending`：任务已接收，等待执行。
- `preparing`：下载或准备工程，建立 `original/` 和 `workspace/`，初始化 Git 基线，执行预编译。
- `generating`：Agent 修改代码。
- `validating`：执行修改后编译验证，生成 `changes.patch`，上传代码包和 diff。
- `completed` / `failed`：上报最终结果。

并发由 `config/config.yaml` 的 `task_manager.max_concurrency` 控制，默认值为 3。超过并发上限的任务进入等待队列。

## 4. 任务目录

每个任务生成独立目录：

```text
results/execution_<id>_<timestamp>/
├── original/
├── workspace/
├── generate/
├── diff/
│   ├── commit.txt
│   └── changes.patch
├── checks/
├── result.json
├── progress_upload_state.json
├── cloud_api_events.json
└── local_execution.log
```

- `original/`：原始工程。
- `workspace/`：Agent 修改后的工程，也是最终打包上传目录。
- `generate/`：Agent 输出、metrics、SSE、HTTP 轮询等运行产物。
- `diff/changes.patch`：基于工作区 Git 基线生成的评审 diff。
- `checks/`：编译验证产物。
- `local_execution.log`：本地完整执行日志。
- `cloud_api_events.json`：云端请求和响应的最近快照。

## 5. 云端上报

进度上报：

```text
POST /api/test-executions/{id}/report
```

最终结果上报：

```text
POST /api/execution-results
```

上报原则：

- 本地保留完整日志。
- 云端 `executionLog` 只上传筛选后的阶段摘要。
- 高频 Agent 日志会做白名单筛选、截断、节流和同秒同类去重。
- 最终结果包含 `isBuildSuccess`、`executionTime`、`tokenConsumption`、`iterationCount`、`outputCodeUrl`、`diffFileUrl` 等字段。

## 6. Agent 与 OpenCode

Agent 配置放在 `config/agents.yaml`。当前主链只使用主修复 Agent，执行后端为 OpenCode。

任务级 OpenCode 隔离：

- 每个任务写入自己的 `workspace/opencode.json`。
- 每个任务挂载自己的 `workspace/.opencode/skills/`。
- 执行器启动 OpenCode 时设置独立 `XDG_CONFIG_HOME`，避免读取用户全局配置。
- HTTP/SSE 请求绑定当前 `workspace`，确保事件和 skill 都归属当前任务。

`question` 工具自动应答逻辑：

- 监听 `question.asked`。
- 默认选择第一个选项。
- 优先以 `question.replied` 或 question `completed` 判断成功。
- SSE 不稳定时用 HTTP 查询兜底。

## 7. 设计原则

- 主流程由代码固定，关键产物路径不交给模型决定。
- `pipeline` 不直接访问云端接口。
- `cloud_api` 不执行 Agent。
- `agent_runner` 只负责单次 Agent 执行。
- `task_manager` 负责任务状态、并发、进度和上报编排。
- 本地日志用于排障，云端日志只展示必要进度。
