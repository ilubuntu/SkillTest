# 本地使用指南

当前版本：`0.0.2`

## 1. 适用场景

用于本地开发机手动启动执行器、触发任务、查看日志和验证 skill 隔离。

## 2. 启动

### macOS / Linux

```bash
./deploy.sh restart
```

### Windows

```bat
deploy.bat restart
```

说明：

- 启动脚本只负责启动 Executor
- OpenCode 由 Python 程序自动检查和拉起
- HarmonyOS 工具链也在启动阶段自动校验

## 3. 健康检查

```bash
curl -s http://127.0.0.1:8000/api/health
curl -s http://127.0.0.1:4096/global/health
```

## 4. 本地 OpenCode 配置

本地跑任务前，建议先确认 `~/.opencode/opencode.json` 可用，至少要把：

- `plugin`
- `provider`
- `permission`
- `agent`

这几部分配好。

一个可参考的最小结构如下：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": [
    "/Users/bb/work/yutouGroup/code/harmony-code"
  ],
  "permission": "allow",
  "agent": {
    "harmonyos-plugin": {
      "permission": "allow"
    },
    "harmonyos-dev": {
      "permission": "allow"
    }
  }
}
```

说明：

- `plugin`
  - 指向本地 OpenCode 插件目录
  - 例如：`/Users/bb/work/yutouGroup/code/harmony-code`
- `permission`
  - 本地调试通常直接设成 `"allow"`，避免工具调用被权限拦住
- `agent`
  - 至少把本地会用到的 agent 配出来
  - 当前常见的是：
    - `harmonyos-plugin`
    - `harmonyos-dev`
  - 这两个 agent 建议都显式加上 `"permission": "allow"`

如果任务依赖本地插件或 agent，但这里没配好，常见现象是：

- Agent 能创建 session，但执行过程没有有效工具行为
- skill / plugin 看起来“挂载了”，但实际不可见或不可调用
- 本地和云端行为不一致

## 5. 常用接口

### 5.1 本地直接跑已有 case

```bash
curl -s -X POST http://127.0.0.1:8000/api/local/run-case \
  -H 'Content-Type: application/json' \
  -d '{
    "caseDir": "/Users/bb/work/benchmark/github/agent_bench/test_cases/bug_fix/002",
    "agentId": "baseline"
  }'
```

可替换的 `agentId`：

- `baseline`
- `baseline-minimax`
- `harmonyos-plugin`
- `harmonyos-plugin-minimax`

### 5.2 云测动态指定 agent

执行器当前支持动态 agent 路由：

```bash
curl -s -X POST http://127.0.0.1:8000/api/cloud-api/agent/baseline-minimax \
  -H 'Content-Type: application/json' \
  -d '{
    "executionId": 10001,
    "cloudBaseUrl": "http://127.0.0.1:3000",
    "agentId": "baseline-minimax",
    "testCase": {
      "input": "修复问题",
      "expectedOutput": "",
      "fileUrl": "http://example.com/test-case.zip"
    }
  }'
```

也兼容这个写法：

```bash
http://127.0.0.1:8000/api/cloud-api/agent/id=baseline-minimax
```

推荐统一使用：

- `/api/cloud-api/agent/{agent_id}`

当前可用的 4 个 `agentId`：

- `baseline`
- `baseline-minimax`
- `harmonyos-plugin`
- `harmonyos-plugin-minimax`

### 5.3 查看本地任务状态

```bash
curl -s "http://127.0.0.1:8000/api/local/status?execution_id=<执行ID>"
```

### 5.4 查看云测任务状态

```bash
curl -s "http://127.0.0.1:8000/api/cloud-api/status?execution_id=<执行ID>"
```

## 6. 看日志

### 6.1 进程级日志

```bash
tail -f logs/agent_bench.log
```

用途：

- 看服务启动
- 看主状态
- 看错误摘要

### 6.2 单任务详细日志

```bash
tail -f results/execution_<id>_<timestamp>/local_execution.log
```

用途：

- 看单个任务完整过程
- 看 Agent 详细行为
- 看上传和上报摘要

## 7. 看任务产物

典型目录：

```text
results/execution_<id>_<timestamp>/
├── original/
├── workspace/
├── generate/
├── diff/
├── checks/
├── local_execution.log
└── cloud_api_events.json
```

重点文件：

- `diff/changes.patch`
  - 最终 patch
- `checks/pre_compile_check/compile.log.txt`
  - 预编译日志
- `checks/post_compile_check/compile.log.txt`
  - 后编译日志
- `generate/output.txt`
  - 主修复 agent 输出

### 7.1 SSE 相关文件

`generate/` 目录下通常还会有：

- `agent_opencode_sse_full.jsonl`
  - 当前 session 的关键原始 SSE 视图
- `agent_opencode_sse_events.jsonl`
  - 当前 session 的裁剪事件流
- `agent_opencode_progress_events.jsonl`
  - 面向日志展示的进度摘要

默认 `sse_filter` 是 `medium`，配置在 [config/config.yaml](/Users/bb/work/benchmark/github/config/config.yaml:47)。

Agent 默认超时当前是 `600s`；如果 `agents.yaml` 里显式配置了 `timeout`，以配置值为准。

当前三档含义：

- `full`
  - 保留更多原始字段，便于排查 SSE 细节
- `medium`
  - 默认档位；保留主要事件，同时去掉 `event/project/time/id/sessionID` 等冗余字段
- `low`
  - 只保留关键事件和最小必要字段，文件最小，但不适合做细粒度 SSE 排查

### 7.2 超时判断

当前超时判断不依赖 SSE 文件是否落盘，而是依赖内存里的运行时活动状态。

活动状态会被两条链路刷新：

- 当前 session 的 SSE 事件
- HTTP `session/message` 轮询拿到的新进展消息

所以即使 SSE 文件为空，只要消息轮询还能持续拿到新的 `reasoning/tool/text` 进展，就不会因为“没写 SSE 文件”直接误判超时。

## 8. 本地验证 skill 隔离

当前实现是：

- 单个共享 OpenCode 服务
- 每个任务自己的 `workspace/opencode.json`
- 每个任务自己的 `workspace/.opencode/skills`

如果要验证任务实际看到了哪些 skill，最直接的方法是让 agent 在输出里打印当前可见 skill 列表，或者直接检查任务目录下：

- `workspace/opencode.json`
- `workspace/.opencode/skills/`

## 9. 本地排查建议

### 9.1 预编译失败

先看：

- `checks/pre_compile_check/compile.log.txt`

这通常能直接区分：

- 工具链问题
- 工程自身编译问题

### 9.2 Agent 没有实际修改代码

看：

- `diff/changes.patch`
- `generate/output.txt`
- `local_execution.log`

### 9.3 上报云测内容

看：

- `cloud_api_events.json`

这里保留最近一次：

- 进度上报请求/响应
- 最终结果上报请求/响应
