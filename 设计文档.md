# Agent Bench — 评测系统设计文档

## 1. 项目背景

团队正在为鸿蒙（HarmonyOS）ArkTS 开发编写一系列**增强配置**（Skill、MCP Tool、System Prompt 等），用于挂载到 AI 编程 Agent（如 OpenCode）上，提升 Agent 的鸿蒙 ArkTS 编程能力。

**核心问题**：如何客观量化这些增强配置的增益价值？

**解决方案**：构建一个**评测系统（Agent Bench）**，通过标准化测试用例，对比 Agent 加载增强配置前后的代码输出质量，生成增益报告。

## 2. 系统定位

- 评测系统是**Web 平台 + CLI 工具**（Python 后端 + 前端），支持可视化操作和命令行自动化两种模式
- **评测对象是 Agent 内部工具**（Skill、MCP Tool、System Prompt 及其组合）
- **Agent 无关**：评测系统通过抽象的 `AgentAdapter` 接口与 Agent 交互，不与任何特定 Agent（如 OpenCode）强绑定，后续可接入 Cursor、Claude Code 等任意 Agent
- Agent 被视为黑盒执行引擎：评测系统只管配置增强工具、发任务、收结果，不介入 Agent 内部执行
- 首版本以 OpenCode 为参考实现，但所有设计和接口均面向 Agent 无关的抽象层

## 3. 架构概览

系统采用**前后端分离 + 三阶段流水线**架构。Web UI 提供可视化操作界面，CLI 提供命令行自动化入口，两者共享同一套 Pipeline Engine。阶段内并行、阶段间串行。

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│ Web UI                                                                               │
│                                                                                      │
│ ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐  ┌────────────┐          │
│ │  用例管理   │ │  Profile   │  │ 评分标准    │ │  评测中心   │  │  报告展示   │          │
│ │  设计/导入  │ │  配置管理   │  │ 管理       │ │  运行/监控  │   │  评分/对比  │          │
│ └────────────┘ └────────────┘ └────────────┘ └────────────┘  └────────────┘          │
│                                                                                      │
└──────────────────────────────────────┬───────────────────────────────────────────────┘
                                       │ REST API + WebSocket
┌──────────────────────────────────────▼───────────────────────────────────────────────┐
│ Backend（FastAPI）                                                                    │
│                                                                                      │
│  ┌─────────────┐   ┌──────────────────────────────────────────────────────────┐      │
│  │             │   │            Pipeline Engine（异步任务引擎）                  │      │
│  │   CLI 入口   │──▶│                                                          │      │
│  │   cli.py    │   │   ┌──────────┐   ┌───────────┐   ┌──────────┐           │      │
│  │             │   │   │  Runner  │──▶│ Evaluator │──▶│ Reporter │           │      │
│  └─────────────┘   │   │ 并行跑用例│   │  并行评分  │   │ 汇总报告  │           │      │
│                     │   └──────────┘   └───────────┘   └──────────┘           │      │
│                     └──────────────────────────────────────────────────────────┘      │
│                                                                                      │
└──────────────────────────────────────┬───────────────────────────────────────────────┘

┌──────────────────────────────────────▼───────────────────────────────────────────────┐

```

### 3.1 核心设计原则

1. **前后端分离** — Web UI 和 CLI 共享同一套 Pipeline Engine，两种入口能力对等
2. **Agent 黑盒** — 评测系统不介入 Agent 内部执行，只管输入/输出
3. **配置驱动** — 通过 Profile 定义增强工具组合，Runner 通过 AgentAdapter.setup() 注入配置
4. **阶段解耦** — Runner、Evaluator、Reporter 三阶段可独立运行，失败后可单独重跑
5. **结果持久化** — 每一步中间结果落盘，支持断点续跑

## 4. 数据管理

系统采用**集中式索引 + 用例目录详细协议**的方式管理测试用例。`test_cases.yaml` 只负责场景与用例索引，单个 case 的详细执行协议放在各自目录下的 `case.yaml` 中。

### 4.1 测试用例注册表（test_cases.yaml）

所有场景和用例的基础元数据集中管理在 `test_cases/test_cases.yaml` 中。该文件只负责：

- 场景发现与筛选
- 用例基本信息展示
- 通过 `case_dir` 定位到具体用例目录

不再在总表中维护 `input_file` / `expected_file` 这类详细执行字段。

```yaml
# test_cases/test_cases.yaml
scenarios:
  - id: scenario_001
    name: "Bug Fix"
    description: "诊断并修复ArkTS代码中的常见缺陷"
    difficulty: medium
    tags: [arkts, bug-fix]
    cases:
      - id: case_001
        case_id: bug_fix_001
        title: "商品列表与详情编辑工程缺陷修复"
        case_dir: test_cases/bug_fix/001
        prompt: "这是一个已有的 HarmonyOS ArkTS 商品管理工程，请基于现有工程修复缺陷。"

  - id: scenario_002
    name: "Code Compile"
    ...
```

### 4.2 单用例协议（case.yaml）

每个 case 目录下放置一个 `case.yaml`，用于定义这个 case 的详细执行协议。它是系统生成 Agent 任务描述、定位模板工程、组织评分上下文的主要依据。

当前采用最小可用结构：

```yaml
schema: case-spec/v1
version: 1

case:
  id: bug_fix_001
  scenario: bug_fix
  title: 商品列表与详情编辑工程缺陷修复

project:
  type: HarmonyOS ArkTS
  project_dir: original_project
  summary: >
    这是一个商品管理工程。首页展示商品列表，
    点击添加或编辑进入详情页，保存后返回首页并更新列表。

problem:
  summary: >
    当前工程存在功能缺陷、错误用法以及可能的废弃 API / 可编译性问题。
  expected_result:
    - 新增商品后返回首页并更新列表
    - 编辑商品后返回首页并更新列表
    - 异常场景有基本提示

agent:
  purpose: 测试 agent 在已有工程中直接修改代码并完成修复的能力
  work_mode: 直接在当前工程中修改代码
  output_requirements:
    - 修改了哪些文件
    - 每个文件的主要修改内容
    - 修复后达到了什么效果

constraints:
  - 保持现有业务流程不变
  - 优先最小必要修改
  - 不修改签名配置和构建配置
```

**用例目录结构**：

```
test_cases/
├── test_cases.yaml              # 测试用例总表（索引层）
├── bug_fix/
│   └── 001/
│       ├── case.yaml            # 单用例详细协议
│       ├── input.ets            # 约定输入代码（可选）
│       ├── expected.ets         # 约定参考代码（可选）
│       └── original_project/    # 该用例自己的鸿蒙工程模板
├── compilable/
│   ├── 001/
│   └── 002/
```

### 4.3 增强配置注册表（enhancements.yaml）

所有可复用的增强模块（Skill、System Prompt、MCP Server）集中管理在 `enhancements/enhancements.yaml` 中，文件按类型分目录存放。

```yaml
# enhancements/enhancements.yaml
skills:
  - id: skill_001
    name: bug_fix
    path: enhancements/skills/bug_fix.md
    description: "Bug修复Skill - 修复ArkTS列表性能、状态管理和组件生命周期等常见缺陷"

  - id: skill_005
    name: build-harmony-project
    path: enhancements/skills/build-harmony-project.md
    description: "HarmonyOS编译Skill - 在非DevEco Studio环境中编译HarmonyOS Next应用"

system_prompts:
  - id: sp_001
    name: "问题修复专家"
    path: enhancements/system_prompt/bug_fix.txt
    description: "问题修复场景的系统提示词"

mcp_servers:
  # 待扩展
  []
```

**增强配置目录结构**：

```
enhancements/
├── enhancements.yaml            # 增强配置总表（唯一元数据源）
├── skills/                      # Skill 文件（Prompt 最佳实践）
│   ├── bug_fix.md
│   ├── compilable.md
└── system_prompt/               # System Prompt 文件
    ├── bug_fix.txt
    ├── compilable.txt
```

## 5. Profile（评测配置）

评测对象不是单一的 Skill，而是**任何能增强 Agent 能力的配置组合**，统一抽象为 **Profile**。

### 5.1 增强项类型

| 类型 | 具体形式 | 注入方式 |
|------|---------|---------|
| Skill | Prompt 最佳实践文件（.md） | `enhancements.skills` → adapter.setup() 注入 |
| MCP Tool | MCP Server 配置 | `enhancements.mcp_servers` → adapter.setup() 注册 |
| System Prompt | 系统级指令 | `enhancements.system_prompt` → adapter.setup() 注入 |
| Tools | 工具开关/配置 | `enhancements.tools` → adapter.setup() 配置 |
| 组合 | 以上任意搭配 | 各项同时通过 adapter.setup() 配置 |

> Profile 的增强配置描述的是"要注入什么"，而非"如何注入到某个具体 Agent"。如何注入是 AgentAdapter 各实现的职责。

### 5.2 Profile 配置格式

Profile 通过 ID 引用 `test_cases.yaml` 中的场景和 `enhancements.yaml` 中的增强配置：

```yaml
# profiles/bug_fix_enhanced.yaml
id: profile_002
name: "Bug Fix Enhanced"
description: "Bug修复增强配置：Skill + 定制化最佳实践"
scenario_ids: [scenario_001]         # 引用 test_cases.yaml 中的场景 id
enhancement_ids:                     # 引用 enhancements.yaml 中的增强配置 id
  - skill_001                        # Bug修复 skill
  - sp_001                           # 问题修复专家 system_prompt
```

```yaml
# profiles/baseline.yaml
id: profile_001
name: "Baseline"
description: "裸Agent，无任何增强配置"
scenario_ids: []
enhancement_ids: []
```

### 5.3 Profile、场景与用例的关系

```
Profile                     场景（Scenario）            测试用例（Case）
┌──────────────────┐       ┌───────────┐             ┌───────────────┐
│ profile_002       │       │scenario_  │             │ bug_fix_001   │
│ Bug Fix Enhanced  │──────▶│  001      │◀────────────│ bug_fix_002   │
│                  │       │ Bug Fix   │             │ bug_fix_003   │
│ scenario_ids:    │       └───────────┘             └───────────────┘
│   [scenario_001] │
│                  │
│ enhancement_ids:  │
│   - skill_001    │
│   - sp_001       │
└──────────────────┘

Profile 管"怎么配置"        场景管"测哪类能力"           用例管"测什么、怎么评"
```

| 层级 | 数量关系 | 执行方式 | 职责 |
|------|---------|---------|------|
| Profile | 一次评测选 1 个 | — | 定义增强配置 + 归属场景 |
| Scenario | 1 个 Profile 归属 1~N 个场景 | 场景间**串行** | 按能力类型分类 |
| Case | 每个场景下 1~N 个用例 | 场景内**并行** | 定义工程上下文、问题描述、约束与评分上下文 |

### 5.4 AgentAdapter 抽象接口

所有 Agent 接入评测系统必须实现统一的适配器接口：

```python
class AgentAdapter(ABC):
    def setup(self, enhancements: dict, on_progress=None):
        """配置增强工具（skills/mcp/system_prompt/tools）"""

    def execute(self, prompt: str, tag: str = "") -> str:
        """执行任务，返回 Agent 输出"""

    def teardown(self):
        """清理增强配置，恢复初始状态"""
```

增强配置的注入有两条互补路径：

| 路径 | 适用场景 | 说明 |
|------|---------|------|
| **API 注入** | 纯文本类配置 | 通过 Agent API 的 system/tools 等字段传递 |
| **沙箱文件注入** | 需要文件系统的配置 | Skill 附件、MCP Server 工作目录等 |

### 5.5 上下文隔离策略

| 层面 | 隔离方式 |
|------|---------|
| 对话历史 | 每个用例创建独立会话（Session），无历史污染 |
| 文件系统 | 每个用例使用独立沙箱目录 |
| 增强配置 | 基线运行 adapter.setup({})，增强运行 adapter.setup(enhancements)，每次 teardown 清理 |
| 模型参数 | config.yaml 统一配置，不写入 Profile |

## 6. 评测流程

### 6.1 三阶段流水线

```
阶段1: Runner（并行跑用例）  → 所有 Agent 输出落盘
阶段2: Evaluator（并行评分） → 所有评分结果落盘
阶段3: Reporter（汇总）      → 生成增益报告
```

每个阶段可独立运行，支持失败后单独重跑某个阶段。

**执行流程示例**：

```
用户执行: python3 cli.py run --profile bug_fix_enhanced

1. 加载 Profile → 读取 scenario_ids → 读取增强配置
2. 串行遍历场景
   └── 场景 [Bug Fix]:
       加载 test_cases.yaml 中 scenario_001 的用例索引 → [bug_fix_001, ...]
       并行执行每个用例:
       ├── Case 001: 读取 case.yaml → 加载 original_project → 基线运行 → 增强运行 → 本地评分 → LLM评分
       ├── Case 002:（同上，独立沙箱）
       └── Case 003:（同上，独立沙箱）
3. 汇总所有用例结果 → 持久化 → 生成报告
```

**CLI 用法**：

```bash
# 完整流程
python cli.py run --profile bug_fix_enhanced

# 分阶段运行
python cli.py run-agents --profile bug_fix_enhanced    # 只跑 Agent
python cli.py evaluate --run-id 20260324_153000        # 只评分
python cli.py report --run-id 20260324_153000          # 只出报告

# 基线管理
python cli.py baseline --cases bug_fix/                # 跑基线
python cli.py run --profile bug_fix_enhanced --refresh-baseline  # 强制重跑基线
```

### 6.2 基线预跑 + 缓存

基线结果只跟用例 + Agent + 模型版本有关，与被测增强配置无关，**无需每次评测重跑**。

```
首次运行：  跑基线 → 缓存结果 → 跑增强 → 评分 → 算增益
后续运行：  读基线缓存 → 跑增强 → 评分 → 算增益     （时间减半）
```

失效条件：当 Agent 版本或模型版本变更时，基线需重跑。

### 6.3 沙箱（Sandbox）

每个用例执行时创建独立的沙箱目录，并在阶段目录中保存中间产物：

```
sandbox/{run_id}/{case_id}/        ← 临时目录，跑完可清理
├── original_project/              ← 用例模板工程副本
└── agent_config/                  ← 增强配置（由 adapter 按需生成）

results/{run_id}/cases/{case_id}/  ← 持久目录，长期保存
├── baseline/
│   ├── input.txt                  ← 发给 Agent 的任务描述
│   ├── output.txt                 ← 基线 Agent 输出
│   ├── compile_sandbox/           ← 用于编译验证的工程副本
│   └── changed_files.json         ← 最简版输出观测结果
├── enhanced/
│   ├── input.txt
│   ├── output.txt
│   ├── compile_sandbox/
│   └── changed_files.json
├── rule_check/
│   └── internal_score.json
├── llm_judge/
│   ├── prompt.txt
│   └── judge.json
└── result.json                    ← 综合结果（含 compile_results）
```

### 6.4 编译检查

编译检查不再依赖全局 `empty_hos_project`。每个 case 都有自己的 `original_project`，编译验证基于该 case 的模板工程完成。

- 将当前 case 的 `original_project` 复制到阶段目录下的 `compile_sandbox`
- 按当前过渡实现，将 Agent 输出主代码写入约定入口文件
- 调用 hvigor 编译命令验证整个工程
- 编译结果（pass/fail + 错误信息）存入 `compile_results`
- 同时对 `original_project` 与 `compile_sandbox` 做最简 diff，生成 `changed_files.json`

## 7. 评分体系

评分由两部分协作：**本地评分**负责确定性检查，**LLM Judge**负责综合打分。

```
评分标准（Rubric）──┐
  维度/权重/准则     ├──▶ LLM Judge ──▶ 综合评分
本地评分系统 ────────┘
  规则匹配/编译检查
```

### 7.1 评分占比

| 评分项 | 占比 | 说明 |
|--------|------|------|
| **LLM Judge** | **70%**（70分） | 按 Rubric 维度综合打分，使用独立模型 |
| **本地评分** | **30%**（30分） | 确定性检查（规则匹配 + 本地编译验证） |

**最终评分** = LLM Judge 得分 + 本地评分得分

### 7.2 评分标准（Rubric）

通过 Web UI 的「评分标准管理」模块维护，支持按场景配置。

按"基本质量→工程可信→生态与创新"三层组织，比例 40%/40%/20%。

| 维度 | ID | 权重 | 说明 |
|------|-----|------|------|
| 基本质量 | `basic_quality` | 40% | 组件可编译性、兼容性、基础体验、稳定性等 |
| 工程可信 | `engineering_trust` | 40% | 异常处理、组件规范、代码质量等 |
| 生态与创新 | `ecosystem_innovation` | 20% | 应用或者元服务的生态规则、行业创新能力等 |

**基本质量维度规则**：

| 规则ID | 规则名 | 级别 | 说明 |
|--------|--------|------|------|
| B1 | 类型定义规范 | HIGH | 禁止使用any，接口/类型定义清晰 |
| B2 | 编译通过 | HIGH | 代码可通过ArkTS编译器检查 |

**工程可信维度规则**：

| 规则ID | 规则名 | 级别 | 说明 |
|--------|--------|------|------|
| E1 | 异常处理 | HIGH | 异步操作有.catch()或try/catch |
| E2 | 代码重复度 | MEDIUM | 重复代码已提取为公共函数 |
| E3 | 禁止废弃API | MEDIUM | 禁止router.push等废弃API |

**生态与创新维度规则**：

| 规则ID | 规则名 | 级别 | 说明 |
|--------|--------|------|------|
| G1 | 日志规范 | HIGH | 使用hilog替代console.log |
| G2 | 路由规范 | MEDIUM | 使用Navigation路由 |

### 7.3 本地评分（确定性评分）

本地评分执行可自动化的确定性检查，占总分 30%。它由两部分组成：

- **规则匹配**：针对 Agent 输出代码进行文本规则检查
- **本地编译验证**：对最终工程副本执行 hvigor 编译检查

设计原则：

- 不预先限定 Agent 输出文件个数和结构
- 系统通过观察最终工程结果进行验证，不干扰 Agent 本身的输出策略
- 当前最简观测产物为 `changed_files.json`，用于记录模板工程与最终工程副本之间的变更文件列表

**评分规则**：
- 代码不涉及某规则时，默认认定该规则通过
- 每个维度基础分 = 30分 / 维度数
- 维度内按规则级别分配：HIGH 50% / MEDIUM 30% / LOW 20%
- Pass = 该规则全部分数，Fail = 0分

**评分规则配置**（`config/internal_rules.yaml`）：

```yaml
# 格式：规则名称 | 级别 | 描述 | 匹配规则 | 是否通过
basic_quality:
  - name: 装饰器使用规范
    level: HIGH
    description: 组件应使用 @Component/@Entry 等 ArkTS 装饰器
    pattern: "@Component|@Entry|@State|@Prop|@Link|@Observed|@Watch"
    pass_on_match: true

engineering_trust:
  - name: 异常处理
    level: HIGH
    description: 异步操作应有 .catch() 或 try/catch 进行异常处理
    pattern: "\\.catch\\s*\\(|try\\s*\\{"
    pass_on_match: true

ecosystem_innovation:
  - name: 组件复用能力
    level: LOW
    description: 应正确使用 @Reusable 实现组件复用
    pattern: "@Reusable"
    pass_on_match: true
```

### 7.4 LLM Judge（综合打分）

LLM Judge 接收：

- 由 `case.yaml` 生成的任务说明
- Agent 输出代码
- 评分标准（Rubric）

按维度打分，占总分 70%。

- 使用**独立模型**，与被测 Agent 的 LLM 分离，避免"自己给自己打分"
- 按评分标准中的维度权重计算加权平均分
- 不再依赖 `input_file / expected_file` 作为评分 Prompt 的固定输入
- 当 `only_run_baseline=true` 时，只做基线侧评分，不再生成伪增强评分

```yaml
# config.yaml
judge:
  provider: zhipu
  model: glm-4-plus
  temperature: 0
```

## 8. 报告输出

评测报告同时输出 JSON（机器可读）+ Markdown（人可读），包含：

- **加权总分**：场景评分×80% + 通用评分×20%
- **场景用例汇总**：用例总数、基线/增强平均分、增益、通过率
- **通用用例汇总**：用例总数、基线/增强平均分、增益、通用通过数
- **通用检查**：编译通过率（基线 vs 增强）
- **逐场景汇总**：每个场景的平均基线分、增强分、增益
- **逐用例明细**：每个用例的基线得分、增强得分、增益值、编译结果
- **维度均分对比**：基本质量/工程可信/生态与创新各维度的前后对比
- **异常标记**：增益为负的用例特别标注

**加权总分计算公式**：

```
加权总分 = 场景用例平均分 × 80% + 通用用例平均分 × 20%
```

其中：
- 场景用例平均分：所有非通用场景用例的 `baseline_total/enhanced_total` 平均值
- 通用用例平均分：通用用例的 `baseline_total/enhanced_total` 平均值（100或0）
- 通用检查：独立统计，不参与加权计算，仅用于展示
