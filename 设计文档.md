# Agent Bench — 评测系统设计文档

## 1. 项目背景

团队正在为鸿蒙（HarmonyOS）ArkTS 开发编写一系列**增强配置**（Skill、MCP Tool、System Prompt 等），用于挂载到 AI 编程 Agent（如 OpenCode）上，提升 Agent 的鸿蒙 ArkTS 编程能力。

**核心问题**：如何客观量化这些增强配置的增益价值？

**解决方案**：构建一个**评测系统（Agent Bench）**，通过标准化测试用例，对比 Agent 加载增强配置前后的代码输出质量，生成增益报告。

## 2. 系统定位

- 评测系统是**独立的 CLI 工具**（Python），不是 Skill
- 评测对象是**任何能增强 Agent 能力的配置组合**（Skill、MCP Tool、System Prompt 及其组合），不是 Agent 本身
- Agent 被视为黑盒：评测系统只管发任务、收结果，不介入 Agent 内部执行
- 首版本聚焦最小可行闭环，后续迭代扩展

## 3. 架构概览

系统采用**三阶段流水线**架构，阶段内并行、阶段间串行：

```
┌──────────────────────────────────────────────────────────────────────┐
│ Agent Bench（独立 CLI 工具）                                          │
│                                                                      │
│  ┌──────────┐     ┌───────────┐     ┌──────────┐                    │
│  │ 阶段1     │     │ 阶段2      │     │ 阶段3    │                    │
│  │ Runner   │ ──▶ │ Evaluator │ ──▶ │ Reporter │                    │
│  │ 并行跑用例 │     │ 并行评分   │     │ 汇总报告  │                    │
│  └────┬─────┘     └───────────┘     └──────────┘                    │
│       │                  ↑                                           │
│       │            ┌─────┴──────┐                                    │
│       │            │ LLM Judge  │  ← 独立模型，非被测 Agent 的 LLM     │
│       │            └────────────┘                                    │
│  ┌────▼──────────────────────────────────────────────────────┐      │
│  │ 沙箱目录                                                    │      │
│  │ sandbox/{case_id}/                                         │      │
│  │  ├── opencode.json     ← 由 Profile 生成                   │      │
│  │  ├── .opencode/skills/ ← Skill 文件（增强配置时存在）         │      │
│  │  ├── AGENTS.md         ← System Prompt（增强配置时存在）     │      │
│  │  └── input.ets         ← 用例输入文件                       │      │
│  └───────────────────────────┬───────────────────────────────┘      │
└──────────────────────────────┼───────────────────────────────────────┘
                               │ opencode run（黑盒调用）
┌──────────────────────────────▼───────────────────────────────────────┐
│ Agent 层（OpenCode）                                                  │
│  Agent 配置 → 规划·调度 → 工具层 (MCP Server / 本地工具)               │
└──────────────────────────────┬───────────────────────────────────────┘
                               │ 双向通信
┌──────────────────────────────▼───────────────────────────────────────┐
│ LLM API 层                                                           │
│  推理 · 生成代码 · 工具决策                                            │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.1 核心设计原则

1. **Agent 黑盒** — 评测系统不介入 Agent 内部执行，只管输入/输出
2. **配置驱动** — 通过 Profile 控制 Agent 加载什么，Runner 根据 Profile 生成沙箱配置
3. **阶段解耦** — Runner、Evaluator、Reporter 三阶段可独立运行，失败后可单独重跑
4. **结果持久化** — 每一步中间结果落盘，支持断点续跑
5. **Judge 独立** — LLM 评分使用独立模型，与被测 Agent 的 LLM 分离

## 4. 测试用例体系

用例分为**通用用例**和**场景用例**两层。通用用例是所有被测对象都必须通过的基础门槛，场景用例按能力类型细分，用于量化特定场景的增益。

### 4.1 通用用例

通用用例检验 Agent 输出的基础质量，**所有被测对象都必须跑**：

| 检查项 | 方式 | 结果 |
|--------|------|------|
| 编译检查 | 调用 ArkTS 编译工具链 | pass / fail |
| 代码格式检查 | 静态分析（lint 规则） | pass / fail + 问题列表 |

通用用例不算增益分数，只算**通过率变化**。如果增强配置导致通用用例通过率下降，说明引入了基础质量退化。

### 4.2 场景用例

场景用例按 Agent 的能力类型划分为 7 个场景：

| 场景 | 说明 | 示例任务 |
|------|------|---------|
| **工程生成** | 从零生成项目/模块骨架 | 生成一个 ArkTS List 页面工程、生成 Navigation 路由框架 |
| **需求开发** | 根据需求描述实现功能代码 | 实现下拉刷新列表、实现登录表单验证 |
| **Bug 修复** | 修复已有代码中的缺陷 | List 组件滑动崩溃修复、状态管理 @State 误用修复 |
| **代码重构** | 不改功能，优化代码结构 | 提取公共组件、拆分大函数、状态管理模式重构 |
| **性能优化** | 不改功能，提升运行性能 | 列表渲染优化（ForEach→LazyForEach）、内存泄漏修复、启动耗时优化 |
| **多模态** | 基于图片/设计稿生成代码 | UI 设计图转 ArkTS 组件代码 |
| **测试用例生成** | 为已有代码生成单元测试 | 为组件生成测试代码、为工具函数生成边界测试 |

### 4.3 被测对象与场景的关系

每个被测对象（Skill、MCP Tool、System Prompt 或其组合）按其能力归属到一个或多个场景。评测时：

```
1. 跑通用用例（必跑） → 编译通过？格式合规？ → 不过则标记基础质量问题
2. 跑归属场景用例      → 按维度打分 → 与基线对比算增益
```

Profile 中声明被测对象归属的场景：

```yaml
# profiles/bug_fix_enhanced.yaml
name: "bug_fix_enhanced"
description: "Bug修复增强配置"
scenarios:                          # 归属场景，决定跑哪些场景用例
  - bug_fix
```

```yaml
# profiles/arkts_full.yaml
name: "arkts_full"
description: "ArkTS全能力增强配置"
scenarios:                          # 可归属多个场景
  - project_gen
  - requirement
  - bug_fix
  - refactor
  - performance
```

## 5. 测试对象：Profile（Agent 配置）

评测对象不是单一的 Skill，而是**任何能增强 Agent 能力的配置组合**，统一抽象为 **Profile**。

### 5.1 增强项类型

| 类型 | 具体形式 | 在 OpenCode 中如何生效 |
|------|---------|----------------------|
| Skill | Prompt 文件 | `.opencode/skills/<name>/SKILL.md` |
| MCP Tool | MCP Server 配置 | `opencode.json` 中 `"mcp"` 字段 |
| System Prompt | 系统提示词 | `AGENTS.md` 或 Agent 配置的 `"prompt"` 字段 |
| 组合 | 以上任意搭配 | 各项同时配置 |

### 5.2 Profile 配置格式

每个 Profile 描述一组 Agent 配置，评测时通过 Profile 控制沙箱环境：

```yaml
# profiles/bug_fix_enhanced.yaml
name: "bug_fix_enhanced"
description: "Bug修复增强配置：Skill + MCP工具 + 定制System Prompt"
scenarios:                              # 归属场景，决定跑哪些场景用例
  - bug_fix

enhancements:
  skills:
    - name: bug_fix
      path: skills/bug_fix/SKILL.md
  mcp_servers:
    - name: arkts-tools
      command: "npx"
      args: ["@anthropic-ai/arkts-mcp", "--project", "./"]
  system_prompt: |
    你是一个专注于鸿蒙ArkTS开发的编程助手，擅长诊断和修复代码缺陷...

# Agent 运行参数
agent:
  model: "anthropic/claude-sonnet-4-6"
  timeout: 120  # 单个用例最大执行时间（秒）
```

基线 Profile：

```yaml
# profiles/baseline.yaml
name: "baseline"
description: "裸Agent，无任何增强配置"
enhancements: {}

agent:
  model: "anthropic/claude-sonnet-4-6"
  timeout: 120
```

### 5.3 Profile → 沙箱映射

Runner 根据 Profile 为每个用例生成独立的沙箱目录：

```
sandbox/{run_id}/{case_id}/
├── opencode.json              ← 由 Profile 生成（MCP Server 配置等）
├── .opencode/
│   └── skills/
│       └── bug_fix/
│           └── SKILL.md       ← 从 Profile 指定的路径复制
├── AGENTS.md                  ← Profile 中的 system_prompt 写入
└── input.ets                  ← 从用例复制
```

基线沙箱无 `.opencode/skills/`、无 `AGENTS.md`、`opencode.json` 为空配置。

## 6. Agent 驱动方案

### 6.1 上下文隔离策略

| 层面 | 隔离方式 |
|------|---------|
| 对话历史 | 每个用例启动独立 `opencode run` 进程，天然无历史污染 |
| 文件系统 | 每个用例使用独立沙箱目录 |
| Agent 配置 | 沙箱内放置独立的 `opencode.json` + Skill 文件 + `AGENTS.md` |
| 模型参数 | Profile 中固定 model 版本 |
| 随机性消除 | 每个用例可配置多次运行取平均分 |

### 6.2 驱动方式

**首版本：`opencode run`（CLI 非交互模式）**

```bash
cd sandbox/{run_id}/{case_id}
opencode run "修复以下ArkTS代码..." --format json -q
```

- 每次调用是独立进程，天然隔离
- `--format json` 输出结构化结果，便于解析
- `-q` 静默模式适合脚本自动化

**后续版本：OpenCode SDK 编程控制**

通过 `opencode serve` 启动常驻实例，用 SDK 控制会话，复用 MCP Server 连接，避免冷启动开销。

> 参考文档：
> - OpenCode CLI：https://opencode.ai/docs/cli/
> - OpenCode SDK：https://opencode.ai/docs/sdk/
> - OpenCode Server：https://opencode.ai/docs/server/
> - OpenCode Config：https://opencode.ai/docs/config/

### 6.3 并发控制

```yaml
# config.yaml
concurrency:
  agent_runs: 3     # 同时跑几个 Agent 进程
  llm_judge: 5      # LLM 评分并发数
```

## 7. 评测流程

### 7.1 基线预跑 + 缓存

基线结果只跟用例 + Agent + 模型版本有关，与被测增强配置无关，**无需每次评测重跑**。

```
首次运行：  跑基线 → 缓存结果 → 跑增强 → 评分 → 算增益
后续运行：  读基线缓存 → 跑增强 → 评分 → 算增益     （时间减半）
```

基线缓存结构：

```
baselines/{baseline_id}/
├── meta.json                  # 运行环境：Agent版本、模型版本、时间戳
└── cases/
    └── bug_fix_001/
        ├── output.ets         # Agent 输出
        ├── agent_log.json     # 执行日志
        └── score.json         # 评分结果
```

**失效条件**：当 Agent 版本或模型版本变更时，基线需重跑。`meta.json` 记录版本信息供校验：

```json
{
  "agent": "opencode",
  "agent_version": "0.5.2",
  "model": "anthropic/claude-sonnet-4-6",
  "timestamp": "2026-03-20T10:00:00Z"
}
```

### 7.2 三阶段流水线

```
阶段1: Runner（并行跑用例）  → 所有 Agent 输出落盘
阶段2: Evaluator（并行评分） → 所有评分结果落盘
阶段3: Reporter（汇总）      → 生成增益报告
```

每个阶段可独立运行，支持失败后单独重跑某个阶段。

**CLI 用法**：

```bash
# 完整流程
python cli.py run --profile bug_fix_enhanced --cases bug_fix/

# 分阶段运行
python cli.py run-agents --profile bug_fix_enhanced --cases bug_fix/  # 只跑 Agent
python cli.py evaluate --run-id 20260324_153000                        # 只评分
python cli.py report --run-id 20260324_153000                          # 只出报告

# 基线管理
python cli.py baseline --cases bug_fix/                                # 跑基线
python cli.py run --profile bug_fix_enhanced --refresh-baseline        # 强制重跑基线

# 对比
python cli.py compare --baseline baseline --enhanced bug_fix_enhanced
```

### 7.3 评分维度

| 维度 | 权重(建议) | 说明 |
|------|-----------|------|
| 正确性 | 40% | 代码是否解决了问题 |
| 完整性 | 30% | 是否覆盖边界情况、异常处理 |
| 代码质量 | 30% | 是否符合 ArkTS 最佳实践、可读性 |

### 7.4 评分方式

**规则匹配（确定性）**：

- `must_contain`：输出代码必须包含的关键片段
- `must_not_contain`：输出代码不能包含的反模式
- 结果为 pass/fail，用于快速筛查

**LLM-as-Judge（灵活性）**：

- 基于 rubric 让 LLM 按维度打分（0-10 分）
- 使用**独立模型**，与被测 Agent 的 LLM 分离，避免"自己给自己打分"
- temperature 设为 0，确保评分确定性

```yaml
# config.yaml
judge:
  provider: anthropic
  model: claude-sonnet-4-6
  temperature: 0
```

**编译检查（后续版本）**：调用 ArkTS 编译工具链验证编译通过。

## 8. 测试用例格式

每个用例由一个 YAML 描述文件 + 关联代码文件组成：

```yaml
# test_cases/bug_fix/001.yaml
id: bug_fix_001
skill_type: bug_fix
title: "ArkTS List组件滑动崩溃修复"

input:
  prompt: "以下ArkTS代码在List组件快速滑动时会崩溃，请修复"
  code_file: cases/bug_fix_001/input.ets

expected:
  reference_file: cases/bug_fix_001/expected.ets
  must_contain:
    - "cachedCount"
  must_not_contain:
    - "forEach"
  compile_check: false
  rubric:
    - name: correctness
      weight: 40
      criteria: "是否修复了滑动崩溃问题，根本原因是List组件缺少cachedCount导致快速滑动时组件复用失败"
    - name: completeness
      weight: 30
      criteria: "是否处理了边界情况：空列表、单条数据、超大数据量"
    - name: code_quality
      weight: 30
      criteria: "代码是否符合ArkTS最佳实践，使用LazyForEach而非ForEach，合理设置cachedCount"
```

## 9. 报告输出

评测报告包含以下内容，同时输出 JSON（机器可读）+ Markdown（人可读）：

### 9.1 报告内容

- **通用用例结果**：编译通过率、格式检查通过率（基线 vs 增强）
- **场景用例总览**：用例总数、整体增益百分比
- **逐场景汇总**：每个场景的平均基线分、增强分、增益
- **逐用例明细**：每个用例的基线得分、增强得分、增益值
- **维度均分对比**：正确性/完整性/代码质量各维度的前后对比
- **异常标记**：增益为负的用例特别标注（增强配置可能引入退化）

### 9.2 报告示例

```json
{
  "run_id": "20260324_153000",
  "profile": "bug_fix_enhanced",
  "baseline": "baseline",
  "general": {
    "compile_pass_rate": { "baseline": 0.9, "enhanced": 0.95 },
    "lint_pass_rate": { "baseline": 0.8, "enhanced": 0.85 }
  },
  "summary": {
    "total_cases": 10,
    "baseline_avg_score": 5.2,
    "enhanced_avg_score": 7.8,
    "gain_percentage": 50.0,
    "regression_count": 1
  },
  "by_scenario": {
    "bug_fix": {
      "case_count": 10,
      "baseline_avg": 5.2,
      "enhanced_avg": 7.8,
      "gain": 2.6
    }
  },
  "dimensions": {
    "correctness": { "baseline": 5.0, "enhanced": 8.2, "gain": 3.2 },
    "completeness": { "baseline": 5.5, "enhanced": 7.5, "gain": 2.0 },
    "code_quality": { "baseline": 5.1, "enhanced": 7.3, "gain": 2.2 }
  },
  "cases": [
    {
      "id": "bug_fix_001",
      "scenario": "bug_fix",
      "baseline_score": 4.0,
      "enhanced_score": 8.5,
      "gain": 4.5,
      "regression": false
    }
  ]
}
```

## 10. 结果持久化

每次运行的中间结果和最终报告全部持久化，支持断点续跑和历史回溯：

```
results/
└── {run_id}/                       # 如 20260324_153000
    ├── config_snapshot.yaml        # 本次运行的完整配置快照
    ├── cases/
    │   └── bug_fix_001/
    │       ├── agent_output.ets    # Agent 输出的代码
    │       ├── agent_log.json      # Agent 执行日志（opencode --format json 输出）
    │       ├── rule_score.json     # 规则匹配评分
    │       └── llm_score.json      # LLM Judge 评分
    └── report/
        ├── report.json             # 机器可读报告
        └── report.md               # 人可读报告
```

## 12. 首版本范围（MVP）

- [ ] 独立 CLI 工具，Python 实现
- [ ] 支持 Profile 配置（Skill + MCP Tool + System Prompt 任意组合）
- [ ] 通过 `opencode run` 驱动 Agent 执行用例
- [ ] 基线预跑 + 缓存，评测时只跑增强配置
- [ ] 评分：规则匹配 + LLM-as-Judge（暂不做编译检查）
