# Agent Bench — 评测系统设计文档

## 1. 项目背景

团队正在为鸿蒙（HarmonyOS）ArkTS 开发建设一套面向 AI 编程 Agent 的评测系统。系统既要支持评估**同一个 Agent 在挂载增益配置前后的能力差异**，也要支持评估**两个不同 Agent 在同一批用例下的执行差异**。

**核心问题**：

- 如何客观量化同一 Agent 挂载增益配置前后的收益？
- 如何在同一批 case 上对比两个 Agent 的最终执行质量？

**解决方案**：构建一个**评测系统（Agent Bench）**，通过标准化测试用例，对比两侧执行配置在同一批 case 下的最终输出质量，生成对比报告。

## 2. 系统定位

- 评测系统是**Web 平台 + CLI 工具**（Python 后端 + 前端），支持可视化操作和命令行自动化两种模式
- **评测对象是 Agent 执行结果**，而不是 Agent 内部工具清单本身
- **Agent 无关**：评测系统通过抽象的 `AgentAdapter` 接口与 Agent 交互，不与任何特定 Agent（如 OpenCode）强绑定，后续可接入 Cursor、Claude Code 等任意 Agent
- Agent 被视为黑盒执行引擎：评测系统只管配置增强工具、发任务、收结果，不介入 Agent 内部执行
- 系统统一采用“双侧对比”模型：无论是“Agent A vs Agent B”，还是“同一 Agent baseline vs enhanced”，本质上都是 `side_a` 与 `side_b` 的结果对比
- 首版本以 OpenCode 为参考实现，但所有设计和接口均面向 Agent 无关的抽象层

### 2.1 当前支持的评测模式

1. **Agent 对比模式**
   - 在相同场景、相同 case 下，对比两个不同 Agent 的执行结果
   - 系统不关心两侧 Agent 内部具体挂载了哪些工具，只关心每一侧的执行配置和最终结果
   - 结果展示采用 `Agent A / Agent B` 或用户自定义标签

2. **增益对比模式**
   - 在相同场景、相同 case 下，对比同一个 Agent 挂载增益配置前后的执行结果
   - 这是原有 baseline / enhanced 模式的保留与泛化
   - 结果展示采用 `Baseline / Enhanced`

## 3. 架构概览

系统采用**前后端分离 + 三阶段流水线**架构。Web UI 提供可视化操作界面，CLI 提供命令行自动化入口，两者共享同一套 Pipeline Engine。阶段内并行、阶段间串行。

```

┌──────────────────────────────────────▼───────────────────────────────────────────────┐
│ Backend（FastAPI）                                                                    │
│                                                                                      │
```

### 3.1 核心设计原则

1. **前后端分离** — Web UI 和 CLI 共享同一套 Pipeline Engine，两种入口能力对等
2. **Agent 黑盒** — 评测系统不介入 Agent 内部执行，只管输入/输出
3. **结果导向** — 评测只关注两侧最终输出与编译/评分结果，不要求系统理解 Agent 内部工具决策细节
4. **配置驱动** — Agent、Profile、增强配置都通过结构化配置描述，Runner 只消费运行时配置
5. **阶段解耦** — Runner、Evaluator、Reporter 三阶段可独立运行，失败后可单独重跑
6. **结果持久化** — 每一步中间结果落盘，支持断点续跑

## 4. 数据管理

系统采用**集中式索引 + 用例目录详细协议**的方式管理测试用例。`test_cases.yaml` 只负责场景与用例索引，单个 case 的详细执行协议放在各自目录下的 `case.yaml` 中。

### 4.1 测试用例注册表（test_cases.yaml）

所有场景和用例的基础元数据集中管理在 `test_cases/test_cases.yaml` 中。该文件只负责：

- 场景发现与筛选
- 用例基本信息展示
- 通过 `case_dir` 定位到具体用例目录

不再在总表中维护 `input_file` / `expected_file` 这类详细执行字段。

```yaml
# test_cases/test_cases.yaml
scenarios:
  - id: scenario_001
    name: "Bug Fix"
    description: "诊断并修复ArkTS代码中的常见缺陷"
    difficulty: medium
    tags: [arkts, bug-fix]
    cases:
      - id: case_001
        case_id: bug_fix_001
        title: "商品列表与详情编辑工程缺陷修复"
        case_dir: test_cases/bug_fix/001
        prompt: "这是一个已有的 HarmonyOS ArkTS 商品管理工程，请基于现有工程修复缺陷。"

  - id: scenario_002
    name: "Code Compile"
    ...
```

### 4.2 单用例协议（case.yaml）

每个 case 目录下放置一个 `case.yaml`，用于定义这个 case 的详细执行协议。它是系统生成 Agent 任务描述、定位模板工程、组织评分上下文的主要依据。

当前采用最小可用结构：

```yaml
schema: case-spec/v1
version: 1

case:
  id: bug_fix_001
  scenario: bug_fix
  title: 商品列表与详情编辑工程缺陷修复

project:
  type: HarmonyOS ArkTS
  project_dir: original_project
  summary: >
    这是一个商品管理工程。首页展示商品列表，
    点击添加或编辑进入详情页，保存后返回首页并更新列表。

problem:
  summary: >
    当前工程存在功能缺陷、错误用法以及可能的废弃 API / 可编译性问题。
  expected_result:
    - 新增商品后返回首页并更新列表
    - 编辑商品后返回首页并更新列表
    - 异常场景有基本提示

agent:
  purpose: 测试 agent 在已有工程中直接修改代码并完成修复的能力
  work_mode: 直接在当前工程中修改代码
  output_requirements:
    - 修改了哪些文件
    - 每个文件的主要修改内容
    - 修复后达到了什么效果

constraints:
  - 保持现有业务流程不变
  - 优先最小必要修改
  - 不修改签名配置和构建配置
```

**用例目录结构**：

```
test_cases/
├── test_cases.yaml              # 测试用例总表（索引层）
├── bug_fix/
│   └── 001/
│       ├── case.yaml            # 单用例详细协议
│       ├── input.ets            # 约定输入代码（可选）
│       ├── expected.ets         # 约定参考代码（可选）
│       └── original_project/    # 该用例自己的鸿蒙工程模板
├── compilable/
│   ├── 001/
│   └── 002/
```


## 5. 评测模式与执行配置

系统的核心抽象不再是“某个 Profile 的增益评测”，而是**同一批 case 下，两侧执行结果的对比**。系统内部统一使用 `side_a / side_b` 表示待比较的两侧。

### 5.1 执行侧（Execution Side）

每一侧都是一个独立的执行配置，至少包含：

- `agent_id`：由哪个 Agent 执行
- `label`：报告中展示的名称
- `profile_id`：可选，表示该侧是否附带一组增强配置

示例：

```yaml
side_a:
  agent_id: opencode_local
  label: "OpenCode"
  profile_id: null

side_b:
  agent_id: cursor_local
  label: "Cursor"
  profile_id: null
```

或：

```yaml
side_a:
  agent_id: opencode_local
  label: "Baseline"
  profile_id: baseline

side_b:
  agent_id: opencode_local
  label: "Enhanced"
  profile_id: bug_fix_enhanced
```

### 5.2 两种对比模式

| 模式 | side_a | side_b | 说明 |
|------|--------|--------|------|
| Agent 对比模式 | Agent A | Agent B | 对比两个不同 Agent 在同一批 case 下的执行差异 |
| 增益对比模式 | 同一 Agent 的基础配置 | 同一 Agent 的增强配置 | 对比同一 Agent 在挂载增益配置前后的执行差异 |

说明：

- 两种模式都只关心**最终执行结果**
- 系统不要求记录或分析 Agent 内部到底挂载了哪些工具
- `Profile` 只是执行侧的一种可选配置来源，不再是唯一中心对象

### 5.3 Agent 定义

为支持多 Agent 横向对比，系统需要引入 Agent 注册表。每个 Agent 定义一个可执行目标：

```yaml
# agents/agents.yaml
agents:
  - id: opencode_local
    name: "OpenCode Local"
    adapter: opencode
    api_base: http://127.0.0.1:4096
    model: default

  - id: cursor_local
    name: "Cursor Local"
    adapter: cursor
    api_base: http://127.0.0.1:4100
    model: default
```

系统通过 `agent_id` 找到对应 Agent 的适配器与连接参数，再根据该侧 `profile_id` 组装运行时配置。

### 5.4 Profile 的角色

Profile 仍然存在，但定位调整为：**执行侧的可选增强配置包**，主要用于增益对比模式，也允许在 Agent 对比模式下为某一侧附带特定增强配置。

#### 5.4.1 增强项类型

| 类型 | 具体形式 | 注入方式 |
|------|---------|---------|
| Skill | Prompt 最佳实践文件（.md） | `runtime_config.skills` → adapter.setup() 注入 |
| MCP Tool | MCP Server 配置 | `runtime_config.mcp_servers` → adapter.setup() 注册 |
| System Prompt | 系统级指令 | `runtime_config.system_prompt` → adapter.setup() 注入 |
| Tools | 工具开关/配置 | `runtime_config.tools` → adapter.setup() 配置 |
| 组合 | 以上任意搭配 | 各项同时通过 adapter.setup() 配置 |

> Profile 描述的是“该侧额外要附带什么配置”，而不是“这个 Agent 内部最终会如何决策工具使用”。

#### 5.4.2 Profile 配置格式

```yaml
# profiles/bug_fix_enhanced.yaml
id: profile_002
name: "Bug Fix Enhanced"
description: "Bug修复增强配置：Skill + 定制化最佳实践"
scenario_ids: [scenario_001]
enhancement_ids:
  - skill_001
  - sp_001
```

```yaml
# profiles/baseline.yaml
id: profile_001
name: "Baseline"
description: "无额外增强配置"
scenario_ids: []
enhancement_ids: []
```

### 5.5 关系模型

```
评测任务（Evaluation Task）
┌──────────────────────────────────────────────┐
│ mode: agent_compare | gain_compare           │
│ side_a: agent_id + optional profile_id       │
│ side_b: agent_id + optional profile_id       │
│ scenarios: [...]                             │
│ case_ids: [...]                              │
└──────────────────────────────────────────────┘
                    │
                    ▼
         场景（Scenario） → 用例（Case）
```

| 层级 | 数量关系 | 执行方式 | 职责 |
|------|---------|---------|------|
| Evaluation Task | 一次评测 1 个 | — | 定义对比模式、两侧执行配置、场景与 case 范围 |
| Execution Side | 每次评测固定 2 个 | 同一 case 下两侧对比 | 定义 Agent、Profile 与展示标签 |
| Scenario | 一次评测选 1~N 个 | 场景间串行 | 按能力类型分类 |
| Case | 每个场景下 1~N 个 | 场景内并行 | 定义工程上下文、问题描述、约束与评分上下文 |

### 5.6 AgentAdapter 抽象接口

所有 Agent 接入评测系统必须实现统一的适配器接口：

```python
class AgentAdapter(ABC):
    def setup(self, runtime_config: dict, on_progress=None):
        """配置当前执行侧的运行时参数，可为空"""

    def execute(self, prompt: str, tag: str = "") -> str:
        """执行任务，返回 Agent 输出"""

    def teardown(self):
        """清理当前执行侧的运行时配置，恢复初始状态"""
```

运行时配置的注入有两条互补路径：

| 路径 | 适用场景 | 说明 |
|------|---------|------|
| **API 注入** | 纯文本类配置 | 通过 Agent API 的 system/tools 等字段传递 |
| **沙箱文件注入** | 需要文件系统的配置 | Skill 附件、MCP Server 工作目录等 |

### 5.7 上下文隔离策略

| 层面 | 隔离方式 |
|------|---------|
| 对话历史 | 每个用例创建独立会话（Session），无历史污染 |
| 文件系统 | 每个用例使用独立沙箱目录 |
| 运行时配置 | 每一侧执行前 `setup(runtime_config)`，执行后 `teardown()` |
| 模型参数 | Agent 定义或全局配置统一管理，不写入 case |

## 6. 评测流程

### 6.1 三阶段流水线

```
阶段1: Runner（并行跑用例）  → 两侧 Agent 输出落盘
阶段2: Evaluator（并行评分） → 所有评分结果落盘
阶段3: Reporter（汇总）      → 生成增益报告
```

每个阶段可独立运行，支持失败后单独重跑某个阶段。

**执行流程示例**：

```
用户执行: python3 cli.py run --task task_bug_fix_compare

1. 加载评测任务 → 解析 mode / side_a / side_b / scenarios / case_ids
2. 串行遍历场景
   └── 场景 [Bug Fix]:
       加载 test_cases.yaml 中 scenario_001 的用例索引 → [bug_fix_001, ...]
       并行执行每个用例:
       ├── Case 001: 读取 case.yaml → 加载 original_project → side_a 运行 → side_b 运行 → 本地评分 → LLM评分
       ├── Case 002:（同上，独立沙箱）
       └── Case 003:（同上，独立沙箱）
3. 汇总所有用例结果 → 持久化 → 生成报告
```

**CLI 用法**：

```bash
# 完整流程
python cli.py run --task task_bug_fix_compare

# 分阶段运行
python cli.py run-agents --task task_bug_fix_compare   # 只跑 Agent
python cli.py evaluate --run-id 20260324_153000        # 只评分
python cli.py report --run-id 20260324_153000          # 只出报告

# 单 case 过滤
python cli.py run --task task_bug_fix_compare --case-id bug_fix_001
```

### 6.2 侧结果缓存

系统不再只缓存“基线”，而是按执行侧缓存结果。缓存键应至少包含：

- `case_id`
- `agent_id`
- `profile_id`
- `model`
- `adapter_version`

对于增益对比模式：

```
首次运行：  跑 side_a（baseline）→ 缓存 → 跑 side_b（enhanced）→ 评分 → 算增益
后续运行：  复用已缓存侧结果，只重跑变更侧
```

对于 Agent 对比模式：

```
首次运行：  跑 side_a（Agent A）→ 缓存 → 跑 side_b（Agent B）→ 评分 → 算差异
后续运行：  哪一侧 Agent / 模型 / 配置未变化，就可直接复用该侧缓存
```

### 6.3 沙箱（Sandbox）

每个用例执行时创建独立的沙箱目录，并在阶段目录中保存中间产物：

```
sandbox/{run_id}/{case_id}/        ← 临时目录，跑完可清理
├── original_project/              ← 用例模板工程副本
└── agent_config/                  ← 增强配置（由 adapter 按需生成）

results/{run_id}/cases/{case_id}/  ← 持久目录，长期保存
├── side_a/
│   ├── input.txt                  ← 发给 Agent 的任务描述
│   ├── output.txt                 ← side_a Agent 输出
│   ├── compile_sandbox/           ← 用于编译验证的工程副本
│   └── changed_files.json         ← 最简版输出观测结果
├── side_b/
│   ├── input.txt
│   ├── output.txt
│   ├── compile_sandbox/
│   └── changed_files.json
├── rule_check/
│   └── internal_score.json
├── llm_judge/
│   ├── prompt.txt
│   └── judge.json
└── result.json                    ← 综合结果（含 compile_results）
```

### 6.4 编译检查

编译检查不再依赖全局 `empty_hos_project`。每个 case 都有自己的 `original_project`，编译验证基于该 case 的模板工程完成。

- 将当前 case 的 `original_project` 复制到阶段目录下的 `compile_sandbox`
- 将当前执行侧的输出应用到工程副本
- 调用 hvigor 编译命令验证整个工程
- 编译结果（pass/fail + 错误信息）存入 `compile_results`
- 同时对 `original_project` 与 `compile_sandbox` 做最简 diff，生成 `changed_files.json`

## 7. 评分体系

评分由两部分协作：**本地评分**负责确定性检查，**LLM Judge**负责综合打分。评分输入来自 `side_a` 与 `side_b` 的最终结果；在增益对比模式下，它们分别映射为 `baseline` 与 `enhanced`。

```
评分标准（Rubric）──┐
  维度/权重/准则     ├──▶ LLM Judge ──▶ 综合评分
本地评分系统 ────────┘
  规则匹配/编译检查
```

### 7.1 评分占比

| 评分项 | 占比 | 说明 |
|--------|------|------|
| **LLM Judge** | **70%**（70分） | 按 Rubric 维度综合打分，使用独立模型 |
| **本地评分** | **30%**（30分） | 确定性检查（规则匹配 + 本地编译验证） |

**最终评分** = LLM Judge 得分 + 本地评分得分

### 7.2 评分标准（Rubric）

通过 Web UI 的「评分标准管理」模块维护，支持按场景配置。

按"基本质量→工程可信→生态与创新"三层组织，比例 40%/40%/20%。

| 维度 | ID | 权重 | 说明 |
|------|-----|------|------|
| 基本质量 | `basic_quality` | 40% | 组件可编译性、兼容性、基础体验、稳定性等 |
| 工程可信 | `engineering_trust` | 40% | 异常处理、组件规范、代码质量等 |
| 生态与创新 | `ecosystem_innovation` | 20% | 应用或者元服务的生态规则、行业创新能力等 |

**基本质量维度规则**：

| 规则ID | 规则名 | 级别 | 说明 |
|--------|--------|------|------|
| B1 | EventHub事件订阅释放 | HIGH | eventhub.on订阅后必须在aboutToDisappear中调用eventhub.off释放，否则内存泄露 |
| B2 | EventHub事件订阅完整性 | HIGH | eventhub.emit触发的事件必须有对应的eventhub.on订阅 |
| B3 | 组件API能力检测 | MEDIUM | 使用需要设备能力的API前应进行canIUse能力检测 |
| B4 | 大数据列表懒加载规范 | MEDIUM | 数据量超过20项应使用LazyForEach并设置cachedCount |

**工程可信维度规则**：

| 规则ID | 规则名 | 级别 | 说明 |
|--------|--------|------|------|
| E1 | 路由规范 | HIGH | 应使用Navigation路由，不推荐router.pushUrl |
| E2 | ForEach keyGenerator规范 | HIGH | ForEach第三个参数keyGenerator必填且值不能重复 |
| E3 | 状态管理装饰器规范 | HIGH | V2状态管理需@ObservedV2+@Trace装饰用于UI更新的字段 |
| E4 | 硬编码字符串规范 | MEDIUM | UI文本应提取为常量或资源引用，避免硬编码字符串 |
| E5 | 异步操作规范 | MEDIUM | 异步操作应使用async/await，避免回调地狱 |

**生态与创新维度规则**：

| 规则ID | 规则名 | 级别 | 说明 |
|--------|--------|------|------|
| G1 | UIAbility生命周期规范 | HIGH | UIAbility生命周期回调应正确实现，资源管理规范 |

### 7.3 本地评分（确定性评分）

本地评分执行可自动化的确定性检查，占总分 30%。它由两部分组成：

- **规则匹配**：针对 Agent 输出代码进行文本规则检查
- **本地编译验证**：对最终工程副本执行 hvigor 编译检查

设计原则：

- 不预先限定 Agent 输出文件个数和结构
- 系统通过观察最终工程结果进行验证，不干扰 Agent 本身的输出策略
- 当前最简观测产物为 `changed_files.json`，用于记录模板工程与最终工程副本之间的变更文件列表

**评分规则**：
- 代码不涉及某规则时，默认认定该规则通过
- 每个维度基础分 = 30分 / 维度数
- 维度内按规则级别分配：HIGH 50% / MEDIUM 30% / LOW 20%
- Pass = 该规则全部分数，Fail = 0分

**评分规则配置**（`config/internal_rules.yaml`）：

```yaml
# 格式：规则名称 | 级别 | 描述 | 匹配规则 | 是否通过
basic_quality:
  - name: 装饰器使用规范
    level: HIGH
    description: 组件应使用 @Component/@Entry 等 ArkTS 装饰器
    pattern: "@Component|@Entry|@State|@Prop|@Link|@Observed|@Watch"
    pass_on_match: true

engineering_trust:
  - name: 异常处理
    level: HIGH
    description: 异步操作应有 .catch() 或 try/catch 进行异常处理
    pattern: "\\.catch\\s*\\(|try\\s*\\{"
    pass_on_match: true

ecosystem_innovation:
  - name: 组件复用能力
    level: LOW
    description: 应正确使用 @Reusable 实现组件复用
    pattern: "@Reusable"
    pass_on_match: true
```

### 7.4 LLM Judge（综合打分）

LLM Judge 接收：

- 由 `case.yaml` 生成的任务说明
- 两侧 Agent 输出代码
- 评分标准（Rubric）

按维度打分，占总分 70%。

- 使用**独立模型**，与被测 Agent 的 LLM 分离，避免"自己给自己打分"
- 按评分标准中的维度权重计算加权平均分
- 不再依赖 `input_file / expected_file` 作为评分 Prompt 的固定输入
- 当系统处于“单侧运行”模式时，只对已执行的一侧评分，不生成伪对侧评分

```yaml
# config.yaml
judge:
  provider: zhipu
  model: glm-4-plus
  temperature: 0
```

## 8. 报告输出

评测报告同时输出 JSON（机器可读）+ Markdown（人可读），包含：

- **加权总分**：场景评分×80% + 通用评分×20%
- **场景用例汇总**：用例总数、side_a / side_b 平均分、差异值、通过率
- **通用用例汇总**：用例总数、side_a / side_b 平均分、差异值、通用通过数
- **通用检查**：编译通过率（side_a vs side_b）
- **逐场景汇总**：每个场景的 side_a 分、side_b 分、差异值
- **逐用例明细**：每个用例的 side_a 得分、side_b 得分、差异值、编译结果
- **维度均分对比**：基本质量/工程可信/生态与创新各维度的前后对比
- **异常标记**：增益为负的用例特别标注

展示层说明：

- Agent 对比模式：`side_a / side_b` 显示为 `Agent A / Agent B`
- 增益对比模式：`side_a / side_b` 显示为 `Baseline / Enhanced`

**加权总分计算公式**：

```
加权总分 = 场景用例平均分 × 80% + 通用用例平均分 × 20%
```

其中：
- 场景用例平均分：所有非通用场景用例的 `side_a_total/side_b_total` 平均值
- 通用用例平均分：通用用例的 `side_a_total/side_b_total` 平均值（100或0）
- 通用检查：独立统计，不参与加权计算，仅用于展示
