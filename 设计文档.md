# Agent Bench — 评测系统设计文档

## 1. 项目背景

团队正在为鸿蒙（HarmonyOS）ArkTS 开发编写一系列**增强配置**（Skill、MCP Tool、System Prompt 等），用于挂载到 AI 编程 Agent（如 OpenCode）上，提升 Agent 的鸿蒙 ArkTS 编程能力。

**核心问题**：如何客观量化这些增强配置的增益价值？

**解决方案**：构建一个**评测系统（Agent Bench）**，通过标准化测试用例，对比 Agent 加载增强配置前后的代码输出质量，生成增益报告。

## 2. 系统定位

- 评测系统是**Web 平台 + CLI 工具**（Python 后端 + 前端），支持可视化操作和命令行自动化两种模式
- **评测对象是 Agent 内部工具**（Skill、MCP Tool、System Prompt 及其组合），不是 Agent 本身
- **Agent 无关**：评测系统通过抽象的 `AgentAdapter` 接口与 Agent 交互，不与任何特定 Agent（如 OpenCode）强绑定，后续可接入 Cursor、Claude Code 等任意 Agent
- Agent 被视为黑盒执行引擎：评测系统只管配置增强工具、发任务、收结果，不介入 Agent 内部执行
- 首版本以 OpenCode 为参考实现，但所有设计和接口均面向 Agent 无关的抽象层

## 3. 架构概览

系统采用**前后端分离 + 三阶段流水线**架构。Web UI 提供可视化操作界面，CLI 提供命令行自动化入口，两者共享同一套 Pipeline Engine。阶段内并行、阶段间串行。

### 3.1 Web UI 模块说明

| 模块 | 功能 |
|------|------|
| **用例管理** | 在线设计用例（prompt、输入代码、评分 rubric）；批量导入（YAML + 代码 zip 包）；按场景分类筛选、预览、编辑 |
| **Profile 配置管理** | 可视化编辑 Profile（选择 Skill、MCP Tool、System Prompt 组合）；基线 Profile 标记；Profile 对比视图 |
| **评分标准管理** | 维护评分维度、权重、Rubric 描述；|
| **评测中心** | 选择 Profile + 用例范围一键发起评测；实时显示每个用例执行状态；WebSocket 推送进度和日志；支持取消运行、失败重跑 |
| **报告展示** | 单次评测增益报告；多次评测趋势对比；导出 JSON / Markdown |

### 3.2 核心设计原则

1. **前后端分离** — Web UI 和 CLI 共享同一套 Pipeline Engine，两种入口能力对等
2. **Agent 黑盒** — 评测系统不介入 Agent 内部执行，只管输入/输出
3. **配置驱动** — 通过 Profile 定义增强工具组合，Runner 通过 AgentAdapter.setup() 注入配置（API 注入 + 沙箱文件互补）
4. **阶段解耦** — Runner、Evaluator、Reporter 三阶段可独立运行，失败后可单独重跑
5. **结果持久化** — 每一步中间结果落盘，支持断点续跑
6. **Judge 独立（可选）** — LLM 评分使用独立模型，与被测 Agent 的 LLM 分离

## 4. 测试用例体系

用例分为**通用用例**和**场景用例**两层。通用用例是所有被测对象都必须通过的基础门槛，场景用例按能力类型细分，用于量化特定场景的增益。

### 4.1 通用用例

通用用例检验 Agent 输出的基础质量，**所有被测对象都必须跑**：

| 检查项 | 方式 | 结果 |
|--------|------|------|
| 编译检查 | 调用 hvigor 编译工具链 | pass / fail + 错误信息 |
| 代码格式检查 | 静态分析（lint 规则） | pass / fail + 问题列表 |

通用用例不算增益分数，只算**通过率变化**。如果增强配置导致通用用例通过率下降，说明引入了基础质量退化。

**编译检查实现**：
- 使用 `empty_hos_project` 模板工程，将 Agent 生成的代码写入 `entry/src/main/ets/pages/Index.ets`
- 调用 hvigor 编译命令进行编译
- 编译结果（pass/fail）存入 `compile_results`，失败时记录错误信息供前端 tooltip 展示

### 4.2 场景用例

场景用例按 Agent 的能力类型划分为 7 个场景：

| 场景 | 说明 | 示例任务 |
|------|------|---------|
| **工程生成** | 从零生成项目/模块骨架 | 生成一个 ArkTS List 页面工程、生成 Navigation 路由框架 |
| **需求开发** | 根据需求描述实现功能代码 | 实现下拉刷新列表、实现登录表单验证 |
| **Bug修复** | 修复已有代码中的缺陷 | List 组件滑动崩溃修复、状态管理 @State 误用修复 |
| **代码重构** | 不改功能，优化代码结构 | 提取公共组件、拆分大函数、状态管理模式重构 |
| **性能优化** | 不改功能，提升运行性能 | 列表渲染优化（ForEach→LazyForEach）、内存泄漏修复、启动耗时优化 |
| **多模态**   | 基于图片/设计稿生成代码 | UI 设计图转 ArkTS 组件代码 |
| **UT生成** | 为已有代码生成单元测试 | 为组件生成测试代码、为工具函数生成边界测试 |

### 4.3 被测对象与场景的关系

每个被测对象（Skill、MCP Tool、System Prompt 或其组合）按其能力归属到一个或多个场景。评测时：

```
1. 跑通用用例（必跑） → 编译通过？格式合规？ → 不过则标记基础质量问题
2. 跑归属场景用例      → 按维度打分 → 与基线对比算增益
```

Profile 中声明被测对象归属的场景：

```yaml
# profiles/bug_fix_enhanced.yaml
name: "bug_fix_enhanced"
description: "Bug修复增强配置"
scenarios:                          # 归属场景，决定跑哪些场景用例
  - bug_fix
```

```yaml
# profiles/arkts_full.yaml
name: "arkts_full"
description: "ArkTS全能力增强配置"
scenarios:                          # 可归属多个场景
  - project_gen
  - requirement
  - bug_fix
  - refactor
  - performance
```

#### 层级与执行方式

```
Profile（评测配置，一次评测选 1 个）
  │
  │  scenarios 字段声明归属场景
  │
  ├── Scenario A（场景，场景间串行遍历）
  │     ├── Case A-001（用例，场景内并行执行，独立沙箱）
  │     ├── Case A-002
  │     └── Case A-003
  │
  └── Scenario B（场景，场景间串行遍历）
        ├── Case B-001
        └── Case B-002
```

| 层级 | 数量关系 | 执行方式 | 职责 |
|------|---------|---------|------|
| Profile | 一次评测选 1 个 | — | 定义增强配置（Skill / MCP / System Prompt）+ Agent 运行参数 |
| Scenario | 1 个 Profile 归属 1~N 个场景 | 场景间**串行** | 按能力类型分类，对应 `test_cases/<scenario>/` 目录 |
| Case | 每个场景下 1~N 个用例 | 场景内**并行** | 定义输入（prompt + 代码）、评分标准（rubric + 规则）、参考答案 |

#### 执行流程

```
用户执行: python3 cli.py run --profile bug_fix_enhanced

1. 加载 Profile (bug_fix_enhanced.yaml)
   → 读取 scenarios: [bug_fix]
   → 读取增强配置: skills, system_prompt

2. 串行遍历场景
   └── 场景 [bug_fix]:
       加载 test_cases/bug_fix/ 下所有用例 → [001, 002, 003]

       并行执行每个用例:
       ├── Case 001:
       │   ├── 创建沙箱 sandbox/{run_id}/bug_fix_001/
       │   ├── 基线运行（裸 Agent）→ 基线输出
       │   ├── 增强运行（注入 Skill + System Prompt）→ 增强输出
       │   ├── 规则评分（must_contain / must_not_contain）
       │   ├── LLM 评分（按 rubric 维度打分）
       │   └── 计算综合分 + 增益
       ├── Case 002:（同上，独立沙箱）
       └── Case 003:（同上，独立沙箱）

3. 汇总所有用例结果 → 持久化 → 生成报告
```

## 5. 测试对象：Profile（Agent 配置）

评测对象不是单一的 Skill，而是**任何能增强 Agent 能力的配置组合**，统一抽象为 **Profile**。

### 5.1 增强项类型

评测对象是 **Agent 内部工具**（Skill、MCP Tool、System Prompt 等），不与任何特定 Agent 绑定。评测系统通过抽象的 `AgentAdapter` 接口与 Agent 交互，不同 Agent 各自实现适配器。

| 类型 | 具体形式 | 抽象协议 | OpenCode 适配示例 |
|------|---------|---------|------------------|
| Skill | Prompt 最佳实践文件（.md） | `enhancements.skills` → adapter.setup() 注入 | API `system` 字段 / 沙箱 `.opencode/skills/` |
| MCP Tool | MCP Server 配置 | `enhancements.mcp_servers` → adapter.setup() 注册 | API `POST /mcp` / 沙箱 `opencode.json` |
| System Prompt | 系统级指令 | `enhancements.system_prompt` → adapter.setup() 注入 | API `system` 字段 / 沙箱 `AGENTS.md` |
| Tools | 工具开关/配置 | `enhancements.tools` → adapter.setup() 配置 | API `tools` 字段 |
| 组合 | 以上任意搭配 | 各项同时通过 adapter.setup() 配置 | 各项同时生效 |

> **关键设计**：Profile 的 `enhancements` 字段描述的是"要注入什么增强工具"，而非"如何注入到某个具体 Agent"。如何注入是 AgentAdapter 各实现的职责。

### 5.2 AgentAdapter 抽象接口

所有 Agent 接入评测系统必须实现统一的适配器接口：

```python
class AgentAdapter(ABC):
    def setup(self, enhancements: dict, on_progress=None):
        """配置增强工具（skills/mcp/system_prompt/tools）"""

    def execute(self, prompt: str, tag: str = "") -> str:
        """执行任务，返回 Agent 输出"""

    def teardown(self):
        """清理增强配置，恢复初始状态"""
```

增强配置的注入有两条互补路径：

| 路径 | 适用场景 | 说明 |
|------|---------|------|
| **API 注入** | 纯文本类配置 | 通过 Agent API 的 system/tools 等字段传递，无需文件系统 |
| **沙箱文件注入** | 需要文件系统的配置 | Skill 附件、MCP Server 工作目录、Agent 配置文件等 |

不同 Agent 可按自身能力选择路径，两者不冲突。

### 5.3 Profile 配置格式

Profile 定义"测什么增强工具" + "测哪些场景"，**不含 Agent/模型配置**（Agent 和模型由 config.yaml 统一控制）：

```yaml
# profiles/bug_fix_enhanced.yaml
name: "bug_fix_enhanced"
description: "Bug修复增强配置：Skill + MCP工具 + 定制System Prompt"
scenarios:                              # 归属场景，决定跑哪些场景用例
  - bug_fix

enhancements:                           # 增强工具配置（Agent 无关）
  skills:
    - name: bug_fix
      path: skills/bug_fix.md           # Skill 文件路径
      attachments:                      # 可选：Skill 附件（需沙箱）
        - templates/bug_fix_template.ets
  mcp_servers:
    - name: arkts-tools
      command: "npx"
      args: ["@anthropic-ai/arkts-mcp", "--project", "./"]
  system_prompt: |
    你是一个专注于鸿蒙ArkTS开发的编程助手，擅长诊断和修复代码缺陷...
  tools:                                # 可选：工具开关
    read: true
    write: true
```

基线 Profile：

```yaml
# profiles/baseline.yaml
name: "baseline"
description: "裸Agent，无任何增强配置"
scenarios: []
enhancements:
  skills:
    - name: build-harmony-project
      path: skills/build-harmony-project.md
  system_prompt: |
    你是一个专注于代码质量的 ArkTS 开发专家，擅长编写可通过编译器检查的正确代码。
    生成代码时请严格遵循 ArkTS 类型系统和语法规范，确保无编译错误。
```

**通用编译 Skill**：所有 Profile（除 project_gen 外）都自动挂载 `build-harmony-project` skill，确保 Agent 生成的代码能通过编译检查。

```yaml
# profiles/bug_fix_enhanced.yaml（示例）
name: "bug_fix_enhanced"
scenarios:
  - bug_fix
enhancements:
  skills:
    - name: build-harmony-project        # 通用编译 Skill
      path: skills/build-harmony-project.md
    - name: bug_fix                      # 场景特定 Skill
      path: skills/bug_fix.md
```

### 5.4 沙箱（Sandbox）

每个用例执行时创建独立的沙箱目录，用于**用例输入文件**和**Agent 增强配置中需要文件系统的部分**：

```
sandbox/{run_id}/{case_id}/        ← 临时目录，跑完可清理
├── input/                         ← 用例输入文件
│   └── input.ets
└── agent_config/                  ← 增强配置（需文件系统的部分，由 adapter 按需生成）
    ├── skills/                    ← Skill 文件 + 附件
    ├── agent.json                 ← Agent 运行配置（如 MCP Server 注册）
    └── system_prompt.md           ← System Prompt（文件形式）

results/{run_id}/cases/{case_id}/  ← 持久目录，长期保存
├── baseline_output.txt            ← 基线 Agent 输出
├── enhanced_output.txt            ← 增强 Agent 输出
├── rule_check.json                ← 规则评分
├── judge.json                     ← LLM 评分
└── result.json                    ← 综合结果
```

sandbox 与 results 职责分离：
- **sandbox**（临时）— 准备运行环境，跑完可清理释放空间
- **results**（持久）— 收集运行结果，长期保存用于报告查看、历史对比、断点续跑

沙箱的职责：
1. **用例输入隔离** — 每个用例的输入文件互不干扰，支持并行执行
2. **增强配置文件** — Skill 附件、MCP Server 工作目录等必须通过文件系统提供的配置

> **注意**：`agent_config/` 目录结构由各 AgentAdapter 实现自行决定。上面示例是通用布局，具体 Agent（如 OpenCode）可映射为自己的配置格式（`.opencode/skills/`、`opencode.json`、`AGENTS.md`）。评测系统不感知 Agent 特定的配置文件格式。

基线沙箱只有 `input/`，无 `agent_config/`。

### 5.5 空项目模板（empty_hos_project）

## 6. Agent 驱动方案

### 6.1 Agent 无关架构

评测系统**不与任何特定 Agent 绑定**。被测对象是 Agent 的内部工具（Skill、MCP Tool、System Prompt），Agent 本身是可替换的执行引擎。

```
评测系统  ──→  AgentAdapter（抽象接口）  ──→  具体 Agent
                    │
          ┌────────┼────────┐
          ▼        ▼        ▼
     OpenCode   Cursor   Claude Code
     Adapter    Adapter   Adapter
```

所有 Agent 接入只需实现 `AgentAdapter` 接口（setup / execute / teardown），评测系统的 Pipeline、Evaluator、Reporter 完全不感知底层 Agent 是谁。

### 6.2 上下文隔离策略

| 层面 | 隔离方式 |
|------|---------|
| 对话历史 | 每个用例创建独立会话（Session），无历史污染 |
| 文件系统 | 每个用例使用独立沙箱目录 |
| 增强配置 | 基线运行 adapter.setup({})，增强运行 adapter.setup(enhancements)，每次 teardown 清理 |
| 模型参数 | config.yaml 统一配置，不写入 Profile |
| 随机性消除 | 每个用例可配置多次运行取平均分 |

### 6.3 驱动方式

通过 `AgentAdapter` 抽象接口驱动 Agent，具体通信方式由各适配器实现决定：

| 适配器 | 通信方式 | 增强注入 |
|--------|---------|---------|
| OpenCodeAdapter | HTTP API（opencode serve） | API 字段（system/tools） + 沙箱文件（skills 附件、MCP 工作目录） |
| 未来：其他 Agent | 各 Agent 的 API/SDK/CLI | 由适配器实现映射 |

**当前实现（OpenCode Server）：**

```
adapter.setup(enhancements)
  → system_prompt + skills → API message.system 字段
  → mcp_servers → API POST /mcp 注册
  → tools → API message.tools 字段
  → skills 附件 → 沙箱文件系统

adapter.execute(prompt)
  → POST /session 创建会话
  → POST /session/{id}/message 发送任务 + 接收结果

adapter.teardown()
  → 清理内存状态 + 沙箱文件
```

### 6.4 并发控制

```yaml
# config.yaml
concurrency:
  agent_runs: 3     # 同时跑几个 Agent 进程
  llm_judge: 5      # LLM 评分并发数
```

## 7. 评测流程

### 7.1 基线预跑 + 缓存

基线结果只跟用例 + Agent + 模型版本有关，与被测增强配置无关，**无需每次评测重跑**。

```
首次运行：  跑基线 → 缓存结果 → 跑增强 → 评分 → 算增益
后续运行：  读基线缓存 → 跑增强 → 评分 → 算增益     （时间减半）
```

基线缓存结构：

```
baselines/{baseline_id}/
├── meta.json                  # 运行环境：Agent版本、模型版本、时间戳
└── cases/
    └── bug_fix_001/
        ├── output.ets         # Agent 输出
        ├── agent_log.json     # 执行日志
        └── score.json         # 评分结果
```

**失效条件**：当 Agent 版本或模型版本变更时，基线需重跑。`meta.json` 记录版本信息供校验：

```json
{
  "agent": "opencode",
  "agent_version": "0.5.2",
  "model": "zhipu/glm-4-plus",
  "timestamp": "2026-03-20T10:00:00Z"
}
```

### 7.2 三阶段流水线

```
阶段1: Runner（并行跑用例）  → 所有 Agent 输出落盘
阶段2: Evaluator（并行评分） → 所有评分结果落盘
阶段3: Reporter（汇总）      → 生成增益报告
```

每个阶段可独立运行，支持失败后单独重跑某个阶段。

**触发方式**：Web UI 评测中心一键发起，或通过 CLI 命令行触发，两者调用同一套 Pipeline Engine。

**CLI 用法**：

```bash
# 完整流程
python cli.py run --profile bug_fix_enhanced --cases bug_fix/

# 分阶段运行
python cli.py run-agents --profile bug_fix_enhanced --cases bug_fix/  # 只跑 Agent
python cli.py evaluate --run-id 20260324_153000                        # 只评分
python cli.py report --run-id 20260324_153000                          # 只出报告

# 基线管理
python cli.py baseline --cases bug_fix/                                # 跑基线
python cli.py run --profile bug_fix_enhanced --refresh-baseline        # 强制重跑基线

# 对比
python cli.py compare --baseline baseline --enhanced bug_fix_enhanced
```

### 7.3 评分体系

评分由三部分协作完成：**评分标准**定义怎么评，**内部评分系统**做确定性检查，**LLM Judge**做综合打分。Evaluator 并行调度评分，具体评分逻辑由评分体系承担。

```
评分标准（Rubric）──┐
  维度/权重/准则     ├──▶ LLM Judge ──▶ 综合评分
内部评分系统 ────────┘
  规则匹配/编译检查
```

#### 7.3.1 评分标准（Rubric）

评分标准通过 Web UI 的「评分标准管理」模块维护，支持按场景配置评分模板：

| 维度 | 权重(建议) | 说明 |
|------|-----------|------|
| 正确性 | 30% | 代码是否解决了问题 |
| 完整性 | 20% | 是否覆盖边界情况、异常处理 |
| 代码质量 | 20% | 是否符合 ArkTS 最佳实践、可读性 |
| 兼容性 | 20% | 是否符合鸿蒙应用的设计约束、是否包含废弃API、废弃权限声明等 |
| 生态规则符合度 | 10% | 是否符合元服务或应用开发的生态规则 |

评分标准可按需扩展维度和权重，不同场景可使用不同的评分模板。

#### 7.3.2 内部评分系统（确定性评分）

内部评分系统执行可自动化的确定性检查，检查结果与 LLM Judge 结果结合形成最终评分。

**评分规则：**

- 如果代码不涉及某规则（即规则匹配结果为空），默认认定该规则**通过**
- 最终综合评分（基线/增强得分）范围为 **0-100 分**，不超过 100 分

**内部评分系统评分计算方式：**

内部评分系统占总分的 **30%**（30分），计算方式如下：

```
内部评分总分 = 30分
维度数 = 与评分标准（Rubric）维度数一致
每维度满分 = 内部评分总分 / 维度数

示例（假设5个维度，每维度基础分 = 30 / 5 = 6分）：
- 代码质量维度满分 = 6分
- 完整性维度满分 = 6分
- 依此类推...

每维度内规则分数分配（HIGH50%/MEDIUM30%/LOW20%）：
- HIGH级别规则：维度满分 × 50%
- MEDIUM级别规则：维度满分 × 30%
- LOW级别规则：维度满分 × 20%

示例（代码质量维度6分，HIGH规则2条，MEDIUM规则1条）：
- HIGH规则总分：6 × 50% = 3分（每条1.5分）
- MEDIUM规则总分：6 × 30% = 1.8分
- LOW规则总分：6 × 20% = 1.2分

规则得分：
- Pass（通过）= 该规则全部分数
- Fail（不通过）= 0分
```

**互斥原则：**

如果内部评分系统未定义某维度的规则（如未定义某个评分标准维度），则该维度在内部评分系统的得分为满分（维度基础分），LLM Judge 获得该维度的全部评分权。

**工作流程：**

```
┌─────────────────────────────────────────────────────────────────┐
│                         评测执行流程                               │
├─────────────────────────────────────────────────────────────────┤
│  1. Agent 生成 Baseline 输出                                     │
│     └─▶ baseline_output.txt（不挂载 Skill 等工具）                │
│                                                                  │
│  2. Agent 生成 Enhanced 输出                                      │
│     └─▶ enhanced_output.txt（挂载 Skill 等工具）                  │
│                                                                  │
│  3. 内部评分系统检测                                              │
│     ├─▶ 读取 config/internal_rules.yaml 评分规则                  │
│     ├─▶ 对 baseline_output.txt 检测，输出 Pass/Fail              │
│     └─▶ 对 enhanced_output.txt 检测，输出 Pass/Fail              │
│                                                                  │
│  4. 生成评分结果文件                                              │
│     └─▶ <case_dir>/internal_rules.yaml                           │
│         (与 baseline_output.txt、enhanced_output.txt 同目录)     │
└─────────────────────────────────────────────────────────────────┘
```

**评分规则配置（config/internal_rules.yaml）：**

```yaml
# 格式说明：规则名称|规则级别|描述|匹配规则|是否通过
# 规则级别: HIGH(高)、MEDIUM(中)、LOW(低)
# 是否通过: true(匹配到规则为通过)/false(匹配到规则为不通过)

# ==================== 兼容性 ====================
compatibility:
  - name: 导入规范
    level: HIGH
    description: import语句应放在文件顶部
    pattern: "^import\\s+"
    pass_on_match: true

  - name: 类型注解
    level: MEDIUM
    description: 建议添加类型注解，避免使用any
    pattern: ": any\\b"
    pass_on_match: false

# ==================== 生态 ====================
ecosystem:
  - name: 日志规范
    level: HIGH
    description: 应使用hilog替代console.log
    pattern: "console\\.log\\("
    pass_on_match: false

  - name: 资源管理
    level: HIGH
    description: 资源使用后应及时释放
    pattern: "\\.close\\(\\)|\\.release\\(\\)|\\.dispose\\(\\)"
    pass_on_match: true

# ==================== 代码质量 ====================
code_quality:
  - name: 空值检查
    level: HIGH
    description: 应使用可选链和空值合并运算符
    pattern: "\\?\\.|\\?\\?|\\?\\?\\="
    pass_on_match: true

  - name: 注释规范
    level: MEDIUM
    description: 公共API应添加注释
    pattern: "\\/\\*\\*"
    pass_on_match: true

#### 7.3.3 LLM Judge（综合打分）

LLM Judge 接收评分标准 + 内部评分结果 + Agent 输出代码，按维度打分。

**评分占比：**

| 评分项 | 占比 | 分数 |
|--------|------|------|
| **LLM Judge** | **70%** | 70分 |
| **内部评分系统** | **30%** | 30分 |

**最终评分公式：**

```
最终评分 = LLM Judge得分 + 内部评分系统得分
        = LLM Judge得分 + 30分 / 维度数 × 各维度得分
```

**LLM Judge（70%）：**

LLM Judge 按评分标准（Rubric）对 Agent 输出代码进行综合打分，占总分 70%。

- 使用**独立模型**，与被测 Agent 的 LLM 分离，避免"自己给自己打分"
- 按评分标准中的维度权重计算加权平均分

**内部评分系统（30%）：**

内部评分系统按维度规则检查得分，占总分 30%。

- 每个维度基础分 = 30分 / 维度数
- 各维度内按规则级别（HIGH/MEDIUM/LOW）分配分数
- Pass = 全部分数，Fail = 0分

**互斥原则：**

如果存在两者之一无某维度评分标准，则另一方获得该维度满分。例如：
- 内部评分系统未定义代码质量维度 → 内部评分系统在该维度得 6 分（满分，每维度基础分）
- LLM Judge 未定义某维度 → LLM Judge 在该维度得满分

**分阶段建议：**

| 阶段 | 评分标准 | 内部评分 | 代码质量 | 说明 |
|------|---------|---------|---------|------|
| 当前（规则子集） | 60% | 20% | 20% | 内部评分置信度有限，评分标准主导 |
| 规则完善后 | 50% | 30% | 20% | 内部评分可信度提升 |
| 长期（完整规则） | 40% | 40% | 20% | 客观与主观平衡 |

      "regression": false
    }
  ]
}
```

## 10. 结果持久化

每次运行的中间结果和最终报告全部持久化，支持断点续跑和历史回溯：

```
results/
└── {run_id}/                       # 如 20260324_153000
    ├── config_snapshot.yaml        # 本次运行的完整配置快照
    ├── cases/
    │   └── bug_fix_001/
    │       ├── agent_output.ets    # Agent 输出的代码
    │       ├── agent_log.json      # Agent 执行日志（opencode --format json 输出）
    │       ├── rule_score.json     # 规则匹配评分
    │       └── llm_score.json      # LLM Judge 评分
    └── report/
        ├── report.json             # 机器可读报告
        └── report.md               # 人可读报告
```
