# Agent Bench 执行器设计文档

## 1. 目标

当前执行器负责接收云测任务，并在本地完成一条固定主链：

1. 准备原始工程与工作区
2. 主修复 Agent 修改代码
3. 编译验证
4. 上传产物
5. 上报进度与最终结果

当前实现仍是单任务执行。后续目标是在不改变单任务主链的前提下，支持多个任务并发执行。

## 2. 当前架构分层

当前代码按 5 层组织。

### `executor`

服务入口层。

职责：

- 启动 FastAPI
- 暴露任务入口与本地查询接口
- 检查 OpenCode 依赖与服务状态

目录：

- `agent_bench/executor/`

### `cloud_api`

任务管理与云端协议层。

职责：

- 接收云测任务
- 维护 execution 状态
- 节流并上报进度
- 在验证完成后上传 `workspace` 和 `diff`
- 上报最终结果

目录：

- `agent_bench/cloud_api/`

### `pipeline`

单任务主流程层。

职责：

- 准备 `original/` 与 `workspace/`
- 预编译验证
- 调主修复 Agent
- 修改后编译验证
- 生成 `diff/commit.txt` 与 `diff/changes.patch`
- 触发 `artifacts_ready`
- 汇总单任务本地结果

目录：

- `agent_bench/pipeline/`

### `agent_runner`

单次 Agent 执行层。

职责：

- 解析 Agent 配置
- 组装 runtime options
- 检查和挂载 skills
- 创建 adapter
- 执行 OpenCode 会话
- 收集 output、metrics、SSE

说明：

- 当前只保留 OpenCode 一种执行后端，不再保留多 adapter 分支
- 主修复 Agent 使用 `build` 模式

目录：

- `agent_bench/agent_runner/`

### `uploader`

上传适配层。

职责：

- 提供统一上传接口
- 按 provider 创建 uploader
- 当前提供 AGC Cloud Storage 实现

目录：

- `agent_bench/uploader/`

## 3. 当前调用关系

当前主链调用关系：

```text
executor
  -> cloud_api
    -> pipeline
      -> agent_runner (generate)
    -> uploader
```

说明：

- `pipeline` 不直接负责云端接口
- `cloud_api` 不直接执行 Agent
- `agent_runner` 只负责单次 Agent 执行
- `uploader` 只负责上传实现，不参与主流程编排

## 4. 单任务主流程

当前单任务固定按以下阶段运行：

1. `pending`
2. `preparing`
3. `generating`
4. `validating`
5. `completed` / `failed`

阶段说明：

- `pending`
  - 任务已接收，等待执行
- `preparing`
  - 下载工程包
  - 解压
  - 建立 `original/`
  - 建立 `workspace/`
  - 初始化工作区 Git 基线
  - 预编译验证
- `generating`
  - 主修复 Agent 修改代码
- `validating`
  - 修改后编译验证
  - 生成 `diff/commit.txt`
  - 生成 `diff/changes.patch`
  - 通过 `artifacts_ready` 事件通知上传 `workspace` 与 `diff`

## 5. 任务目录结构

每次任务在 `results/execution_<id>_<timestamp>/` 下生成独立目录：

```text
results/execution_<id>_<timestamp>/
├── original/
├── workspace/
├── generate/
├── diff/
│   ├── commit.txt
│   └── changes.patch
├── checks/
├── result.json
├── progress_upload_state.json
├── cloud_api_events.json
└── local_execution.log
```

说明：

- `original/` 是原始工程
- `workspace/` 是主修复 Agent 修改后的工程
- `generate/` 保存主修复 Agent 阶段产物
- `diff/` 保存固定生成的提交基线与 patch 文件
- `checks/` 保存编译检查结果
- `local_execution.log` 是单任务本地详细日志
- `cloud_api_events.json` 只保留最近一次云端请求/响应快照

## 6. Agent 阶段产物

每个 Agent 阶段都会生成：

- `*_output.txt`
- `*_interaction_metrics.json`
- `*_opencode_sse_full.jsonl`
- `*_opencode_sse_events.jsonl`
- `*_opencode_progress_events.jsonl`

用途：

- `*_sse_full.jsonl`
  - 完整原始 SSE，用于排查权限事件、卡死和工具细节
- `*_sse_events.jsonl`
  - 裁剪后的原始事件
- `*_progress_events.jsonl`
  - 用于本地展示与云端进度筛选

## 7. 云端接口

### 7.1 进度上报

`POST /api/test-executions/{id}/report`

请求字段：

- `status`
- `errorMessage`
- `conversation`
- `executionLog`

说明：

- `executionLog` 以字符串形式上报
- 字符串内容是阶段数组
- 每个阶段包含：
  - `stage`
  - `message`
  - `detail`
- detail 会做筛选和节流，不直接上报全部本地日志

### 7.2 最终结果上报

`POST /api/execution-results`

核心字段：

- `isBuildSuccess`
- `executionTime`
- `tokenConsumption`
- `iterationCount`
- `codeQualityScore`
- `expectedOutputScore`
- `outputCodeUrl`
- `diffFileUrl`

其中：

- `outputCodeUrl` 来自 `workspace/` 打包上传
- `diffFileUrl` 来自 `diff/changes.patch` 上传
- 本地日志和云端完成态 detail 只记录最终结果上报请求/响应摘要，不打印完整 response body

### 7.3 本地辅助接口

执行器还提供本地接口用于查看状态：

- `/api/cloud-api/status`

## 8. Agent 配置

Agent 配置统一放在：

- `config/agents.yaml`

当前主链只使用 OpenCode。

每个 Agent 都是独立完整配置，不再使用 `defaults`。常用字段：

- `api_base`
- `opencode_agent`
- `model`
- `timeout`
- `tools`
- `mounted_skills`
- `opencode_proxy`
- `extra_prompt`

其中：

- `opencode_proxy` 是按 Agent 配置的 OpenCode 代理环境
- `extra_prompt` 只负责 Agent 行为约束、skill 调用要求和输出格式要求
- 业务 prompt 由 `pipeline/prompts.py` 负责组装
- `deploy.sh` / `deploy.bat` 会从第一个带 `opencode_proxy` 的 Agent 中读取代理配置启动 OpenCode Server
- 当前主链只使用主修复 Agent

### OpenCode skill 隔离

当前实现保留一个共享的 OpenCode HTTP 服务，但每个任务通过自己的 `workspace/` 隔离本地 skill 配置。

任务级配置文件：

- `workspace/opencode.json`
- `workspace/.opencode/skills/<skill>/SKILL.md`

关键实现结论：

- 仅在 HTTP 请求里传 `directory=workspace` 不足以隔离 skill
- 如果 OpenCode 服务仍读取全局 `~/.config/opencode`，全局 skill 仍会泄漏到任务上下文
- 执行器启动 OpenCode 时会额外设置独立的 `XDG_CONFIG_HOME`
- 这样执行器使用的 OpenCode 服务只读取：
  - 执行器专用配置目录
  - 当前任务 `workspace/` 下的本地 `opencode.json`
  - 当前任务 `workspace/.opencode/skills`

已验证行为：

- `baseline` 任务只能看到 `build-harmony-project`
- `static_code_score_agent` 任务只能看到 `harmonyos-gen-code-evaluator`
- `constraint_score_agent` 任务只能看到 `constraint-score-review`

## 9. 当前实现与后续目标

### 当前实现

- `cloud_api/service.py` 仍按单任务模式管理 execution
- `pipeline` 是单任务主流程
- `agent_runner` 已承担单次 Agent 执行的全部职责
- `uploader` 已抽成接口、工厂和 AGC 实现
- interaction metrics 已按来源拆分：
  - `http`
    - OpenCode HTTP 返回的原始会话和消息数据
  - `derived`
    - 本地程序拼接和计算出的 timing、usage、message、tools 信息

### 后续目标

后续需要支持多个任务并发执行。目标结构：

```text
executor
  -> execution_1 -> pipeline_1 -> agent_runner(generate)
  -> execution_2 -> pipeline_2 -> agent_runner(generate)
  -> execution_3 -> pipeline_3 -> agent_runner(generate)
```

每个 execution 都应拥有独立的：

- pipeline 实例
- 状态
- 进度
- 结果目录
- 上报链

## 10. 设计原则

- `executor` 只负责服务入口与任务管理
- `cloud_api` 只负责云端协议、状态和上报
- `pipeline` 只负责单任务主流程
- `agent_runner` 只负责单次 Agent 执行
- `uploader` 只负责上传实现
- 关键产物路径由代码固定，不让模型自由决定
- 本地保留完整日志，云端只上报必要信息
