# Agent Bench — 代码职责划分

## 总体原则

- **职责单一**：每个模块只做一件事，不越界
- **依赖单向**：cli → runner / evaluator / report，模块间不互相依赖
- **通信收口**：所有与外部服务（OpenCode API）的交互集中在 runner 模块，其他模块通过 runner 的公开接口调用
- **不重复造轮子**：HTTP 调用、模型解析等基础能力只写一份，通过 import 复用

## 模块职责

### cli.py — 编排层

| 职责 | 说明 |
|------|------|
| CLI 参数解析 | argparse 定义 --profile、--cases、--dry-run 等参数 |
| 场景解析 | resolve_scenarios()：从 Profile YAML 或 --cases 参数确定要跑哪些场景 |
| 并行调度 | ThreadPoolExecutor 控制场景内用例并行执行 |
| 流程串联 | 按 Runner → Evaluator → Reporter 顺序调用各模块 |
| 进度打印 | 线程安全地输出每个用例的执行日志和最终汇总 |

**不做的事**：不包含任何 HTTP 调用、评分算法、报告渲染逻辑。

### runner/agent_runner.py — Agent 通信层

| 职责 | 说明 |
|------|------|
| HTTP API 调用 | call_http_api()：与 OpenCode 服务通信的唯一入口 |
| 模型解析 | parse_model()：将 "provider/model" 字符串转为 API 格式，全局唯一实现 |
| Prompt 模板 | BASELINE_PROMPT / ENHANCED_PROMPT：基线和增强的 prompt 模板 |
| AgentRunner 类 | run_baseline() / run_enhanced()：构造 prompt 并调用 call_http_api() |
| 服务发现 | ensure_opencode_server()：查找或启动 OpenCode 服务 |
| Sandbox 创建 | create_sandbox()：为每个用例创建隔离的运行环境 |

**关键设计**：call_http_api() 和 parse_model() 是公开函数，evaluator/llm_judge.py 通过 import 复用，避免重复实现。

### evaluator/rule_checker.py — 规则评分

| 职责 | 说明 |
|------|------|
| 确定性评分 | check()：检查 must_contain / must_not_contain 规则，返回通过率得分 |

纯函数，无外部依赖，输入输出确定。

### evaluator/llm_judge.py — LLM 评分

| 职责 | 说明 |
|------|------|
| LLM 评分 | LLMJudge.judge()：构造评审 prompt，调用 LLM 按维度打分 |
| 结果解析 | _parse_scores()：从 LLM 输出中提取 JSON 格式的评分 |

**关键设计**：通过 `from agent_bench.runner.agent_runner import call_http_api` 复用 HTTP 调用，不自己维护通信逻辑。

### report/reporter.py — 报告层

| 职责 | 说明 |
|------|------|
| 汇总计算 | _compute_summary()：计算均分、通过率、维度对比 |
| 场景汇总 | _compute_by_scenario()：按场景分组统计 |
| 报告生成 | generate()：输出 JSON + Markdown 格式报告 |
| Markdown 渲染 | _render_markdown()：渲染可读的报告文档 |

## 依赖关系

```
cli.py（编排层）
  ├── runner/agent_runner.py    ← AgentRunner, ensure_opencode_server
  ├── evaluator/rule_checker.py ← check()
  ├── evaluator/llm_judge.py    ← LLMJudge
  │     └── runner/agent_runner.py ← call_http_api (复用通信)
  └── report/reporter.py        ← generate()
```

## 新增模块的原则

1. 如果涉及与外部 Agent/LLM 服务通信 → 放入 runner/
2. 如果涉及评分逻辑（规则匹配、LLM 评审、编译检查等） → 放入 evaluator/
3. 如果涉及结果汇总和报告输出 → 放入 report/
4. cli.py 只负责参数解析、调度、打印，不膨胀业务逻辑
