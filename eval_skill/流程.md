# 评测 Skill 运行流程详解

## 总体流程图

```
python3 run_eval.py
       │
       ▼
  加载测试用例 (JSON)
  加载业务 Skill (Markdown)
       │
       ▼
  ┌─── 循环每个用例 ───────────────────────────────────┐
  │                                                     │
  │  ① 基线运行: claude -p "任务描述+代码"    → 输出A    │
  │  ② 增强运行: claude -p "Skill+任务描述+代码" → 输出B │
  │  ③ 规则评分: 对 A 和 B 做关键词匹配                  │
  │  ④ LLM评分:  claude -p "评分prompt" 对 A/B 打分     │
  │  ⑤ 汇总该用例的分数                                  │
  │                                                     │
  └─────────────────────────────────────────────────────┘
       │
       ▼
  生成报告 (JSON + Markdown)
```

## 逐步详解

### 第一步：入口 `run_eval.py`

```python
cases = load_test_cases("bug_fix")    # 读取 test_cases/bug_fix/*.json
skill_content = load_skill("bug_fix") # 读取 skills/bug_fix.md
```

它做两件事：
- 扫描 `test_cases/bug_fix/` 目录，加载 `001.json`、`002.json`、`003.json`
- 读取 `skills/bug_fix.md` 的全部内容（这就是被测的业务 Skill）

### 第二步：基线运行（无 Skill）

`runner/agent_runner.py` 的 `run_baseline()` 被调用：

```
构造 prompt:
  "你是一个ArkTS开发者。请修复以下代码中的bug。
   ## 任务
   以下ArkTS代码在List组件快速滑动时会崩溃，请修复
   ## 代码
   {input.ets 的内容}
   ## 要求
   只输出修复后的完整代码"

执行:
  subprocess.run(["claude", "-p", 上面这段prompt])
                    │
                    ▼
         Claude Code 启动一个全新进程
         读取 prompt → LLM 推理 → 输出修复后的代码
                    │
                    ▼
         捕获 stdout 作为 baseline_output（输出A）
```

关键点：`claude -p` 是非交互模式，进程启动 → 输出结果 → 退出，没有对话历史。

### 第三步：增强运行（挂载 Skill）

`run_enhanced()` 被调用，唯一的区别是 **prompt 里多了 Skill 内容**：

```
构造 prompt:
  "你是一个ArkTS开发者。请参考以下最佳实践修复代码中的bug。
   ## 最佳实践参考
   {skills/bug_fix.md 的全部内容}      ← 这是唯一的差别
   ## 任务
   以下ArkTS代码在List组件快速滑动时会崩溃，请修复
   ## 代码
   {input.ets 的内容}
   ## 要求
   只输出修复后的完整代码"

执行:
  subprocess.run(["claude", "-p", 上面这段prompt])
                    │
                    ▼
         Claude Code 又启动一个全新进程（跟基线完全独立）
         这次 LLM 看到了 Skill 里的最佳实践指导
                    │
                    ▼
         捕获 stdout 作为 enhanced_output（输出B）
```

**基线和增强的对比本质**：同样的任务、同样的代码，区别只是 prompt 里有没有 Skill 的知识。如果 Skill 写得好，LLM 会参考它产出更高质量的代码。

### 第四步：规则评分

`evaluator/rule_checker.py` 对输出 A 和输出 B 分别做关键词检查：

```
以用例 001 为例:
  must_contain: ["LazyForEach", "cachedCount", "IDataSource"]
  must_not_contain: ["ForEach(this.items"]

对 baseline_output 检查:
  "LazyForEach" in output?    → True  ✓
  "cachedCount" in output?    → True  ✓
  "IDataSource" in output?    → True  ✓
  "ForEach(this.items" not in output? → True  ✓
  → rule_score = 4/4 * 100 = 100.0

对 enhanced_output 做同样的检查
  → rule_score = 100.0
```

规则评分是**确定性的**，不依赖 LLM，跑多少次结果一样。

### 第五步：LLM 评分（LLM-as-Judge）

`evaluator/llm_judge.py` 再次调用 `claude -p`，但这次不是让它写代码，而是**让它当评委打分**：

```
构造评分 prompt:
  "你是一个严格的代码评审专家，请评估以下ArkTS代码的修复质量。

   ## 原始代码（有bug）
   {input.ets}

   ## 参考答案
   {expected.ets}

   ## 待评估代码
   {baseline_output 或 enhanced_output}

   ## 评分维度
   - correctness（权重40%）: 是否将ForEach替换为LazyForEach...
   - completeness（权重30%）: 是否设置了cachedCount...
   - code_quality（权重30%）: 代码是否符合ArkTS最佳实践...

   请对每个维度打分（0-100），返回JSON"

执行:
  subprocess.run(["claude", "-p", 评分prompt])
                    │
                    ▼
  Claude 返回类似:
  {"scores": [
    {"name": "correctness", "score": 85, "reason": "正确使用了LazyForEach..."},
    {"name": "completeness", "score": 72, "reason": "缺少数据更新通知..."},
    {"name": "code_quality", "score": 78, "reason": "结构合理但命名可优化..."}
  ]}
                    │
                    ▼
  用正则提取 JSON，解析为 dict
```

注意：**每个用例要调 4 次 claude -p**：
1. 基线运行（生成代码）
2. 增强运行（生成代码）
3. 基线评分（LLM 打分）
4. 增强评分（LLM 打分）

### 第六步：分数汇总

```python
# 权重: 规则 30% + LLM 70%
total = 0.3 * rule_score + 0.7 * llm_weighted_avg

# llm_weighted_avg 按 rubric 权重计算:
# correctness(40%) * 85 + completeness(30%) * 72 + code_quality(30%) * 78
# = 34 + 21.6 + 23.4 = 79.0

# total = 0.3 * 100 + 0.7 * 79.0 = 30 + 55.3 = 85.3
```

### 第七步：生成报告

`report/reporter.py` 汇总所有用例的结果：

```
输入: 3个用例的分数明细
       │
       ▼
计算: 平均分、通过率、各维度均分、增益值
       │
       ├── report.json  (机器可读，完整数据)
       └── report.md    (人可读，表格化展示)
```

## 一次完整运行的调用链

以 3 个用例为例，总共发生 **12 次 `claude -p` 调用**：

```
用例001: claude -p (基线) → claude -p (增强) → claude -p (评基线) → claude -p (评增强)
用例002: claude -p (基线) → claude -p (增强) → claude -p (评基线) → claude -p (评增强)
用例003: claude -p (基线) → claude -p (增强) → claude -p (评基线) → claude -p (评增强)
```

每次调用都是独立进程，互不影响，上下文天然隔离。
